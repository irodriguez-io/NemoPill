# Environments And DevOps

## How To Use This File

- This file is the canonical home for every build-, CI-, DevOps-, and quality-gate decision that gates M-001 Foundation and informs every subsequent NemoPill release. It records the *decisions* about build / CI tooling and the operational model — the actual `build.gradle.kts` blocks, `.github/workflows/*.yml` files, `.gitleaks.toml`, `_build/hooks/pre-commit` script, and other code or configuration artifacts implementing those decisions are M-001 Foundation deliverables in `_source/` and `_build/`, written by the Developer role under Apply Mode in a future task packet.
- Update this file whenever an environment, pipeline, deployment contract, or quality gate changes. Examples that warrant a revision: `targetSdk` bumped to API 36 once Android 16 is stable; the CI tool migrated off GitHub Actions; the secret-scanner vendor changed; a runbook authored, retired, or renamed; staged rollout discipline reopened after a P0 fix-forward in anger; the encryption-at-rest decision revisited at T-005 file-13 threat-model contextualization; any of the four runbooks in Section 9 transitions from `Stub` to `Authored`.
- Cross-file rule: this file *records* operational decisions; it does not re-decide content owned by other `_context/` files. The Compliance Applicability Matrix and the data-protection commitments live in `_context/05_engineering_quality_security_and_compliance.md`; the milestone schedule and Dependency And Blocker Register live in `_context/07_delivery_plan_and_milestones.md`; the Sensitive personal data taxonomy lives in `_context/06_integrations_data_flow_and_boundaries.md`. Section 10 DevOps Risks And Trade-Offs cross-references each of those rather than mirroring them.

## Environments

NemoPill is an on-device-only Android client (`_context/04_solution_architecture.md` § Solution Architecture) with no `android.permission.INTERNET` and no off-device data transmission (`_context/05_engineering_quality_security_and_compliance.md` § Data Protection; `_context/06_integrations_data_flow_and_boundaries.md` § Data Ownership And Retention). The framework template's `URL Or Endpoint` column therefore does not apply and has been repurposed as **`Artifact / Delivery Channel`**, describing what the Android build is at each tier (build variant + delivery channel) rather than a network address.

## Infrastructure

- **Hosting target** (repurposed from the framework template's "Hosting provider" bullet, which assumes a project-controlled deploy target): End-user Android devices, distributed via Google Play Store. No project-controlled hosting infrastructure. The only externally-managed surface NemoPill uses is the Google Play Console, which acts as a distribution channel rather than a hosting environment in the framework sense.
- **Compute model** (repurposed): Native Android app process on end-user devices. Background work executes via `WorkManager` periodic workers (per `_context/04_solution_architecture.md` § Solution Architecture's split scheduling / notifications design and the `WorkManager` / `AlarmManager` dual-strategy decision) and `AlarmManager`-triggered `BroadcastReceiver` instances. No remote compute.
- **Android SDK targets** (resolves the `minSdk` / `targetSdk` / `compileSdk` deferral from `_context/07_delivery_plan_and_milestones.md` Dependency And Blocker Register):
  - `minSdk = 26` (Android 8 Oreo). Lowest defensible floor: Android 8 introduced the background-broadcast-receiver restrictions that NemoPill's scheduling and notifications pipeline must handle anyway (per `_context/04_solution_architecture.md` § Solution Architecture), so going below API 26 would add pre-Oreo workarounds without meaningfully broader device reach in 2026. Covers ≈95 %+ of Android devices as of M-006 authoring. Accommodates the platform permission ladder pinned in `_context/06_integrations_data_flow_and_boundaries.md` (`BOOT_COMPLETED` API 24+, background-receiver restrictions API 26+, `SCHEDULE_EXACT_ALARM` API 31+, `POST_NOTIFICATIONS` API 33+) with explicit per-API branches inside the platform-adapter layer rather than via a higher floor.
  - `targetSdk = 35` (Android 15). Latest stable Android API as of MVP authoring. Google Play submission rules require `targetSdk` to be within one year of the latest API, so pinning `35` also keeps the M-006 Play Console submission valid; if Android 16 (API 36) ships and stabilizes before M-006, the future Project-Manager-role agent should open a task packet to bump `targetSdk` and `compileSdk` together rather than letting drift accumulate.
  - `compileSdk = 35` (Android 15). Matches `targetSdk` per standard Android Gradle plugin practice. Re-pinning happens whenever `targetSdk` re-pins.
- **Datastores in use:** **Room** (AndroidX SQLite ORM), pinned at `androidx.room:room-runtime:2.7.0`, `androidx.room:room-compiler:2.7.0`, and `androidx.room:room-ktx:2.7.0` (latest stable in the AndroidX Room 2.7.x track as of MVP authoring). Single Room database per `_context/04_solution_architecture.md` § Solution Architecture (the "Single Room database" decision); the database file lives in the app's private internal storage and is removed by Android when the user uninstalls the app, satisfying the `_context/06_integrations_data_flow_and_boundaries.md` § Data Ownership And Retention "uninstall = total deletion" commitment. No other datastore — no cache, queue, or object store; in-memory state is held by `:application`-layer use-cases per `_context/04_solution_architecture.md`.
- **Network topology:** **Not applicable.** NemoPill is on-device-only and does not request `android.permission.INTERNET` (per `_context/05_engineering_quality_security_and_compliance.md` § Data Protection); there is no network surface to describe.
- **Infrastructure as code tool:** **Not applicable.** No project-controlled infrastructure to provision. The only externally-managed surfaces are the Google Play Console (configured via the Play Console web UI, not declaratively) and the CI runner itself (configured via the CI workflow file referenced in Section 4 Continuous Integration). Neither warrants Terraform, Pulumi, CDK, OpenTofu, or Helm.
- **IaC repository location:** **Not applicable** — see the "Infrastructure as code tool" bullet above.

- **Secret rotation cadence:** Rotate the Android upload key **only on compromise** (per the Secrets row above's rationale — Play App Signing isolates the user-facing signing key from the upload key, so time-based rotation is unnecessary and introduces avoidable risk of self-lockout). Owner = Isidro (solo human gatekeeper). No service-account credentials, no API keys, and no embedded passphrases exist in MVP, so the rotation rule has exactly one subject.
- **Secret-in-code rule:** **Never commit secrets.** Specifically: no `.jks`, `.keystore`, or `.p12` file; no `KEYSTORE_PASSWORD=`, `KEY_PASSWORD=`, or `KEY_ALIAS=` line in `local.properties` or any Gradle file; no API key, OAuth token, or service-account JSON anywhere in `_source/` (there are none in MVP by design — see `_context/05_engineering_quality_security_and_compliance.md` § Secret management — but the rule applies forward-looking). A pre-commit secret-scanner (vendor pinned in Section 4 Continuous Integration) enforces this on every commit and fails the commit if a credential-shaped string or known-bad filename is detected.

## Continuous Integration

- **CI tool:** **GitHub Actions.** First-class native integration with GitHub (the project's repository host), free unlimited minutes for public repos and 2,000 free minutes/month for private repos (more than sufficient for solo-dev velocity on a Robolectric-only test suite targeting a ≤ 10-minute pipeline), and de-facto standard for Android open-source projects with mature Android-specific actions (`actions/setup-java`, `android-actions/setup-android`, `gradle/actions/setup-gradle`). Alternatives (GitLab CI, CircleCI, Buildkite, Jenkins) all require platform migration or self-hosting and do not earn their cost at this scale.

- **Pipeline definition location:**
  - `.github/workflows/ci.yml` — the merge-blocking pipeline; triggers on every push to `main` and every pull request targeting `main`. All eight stages enumerated below run here.
  - `.github/workflows/release.yml` — the M-006 manual-promote release pipeline; triggers only on `workflow_dispatch` (manually run by Isidro from the GitHub UI when promoting a Play Console Internal Testing release to Production). Out of MVP merge-blocking scope; documented in Section 6 Release Process.

- **Required CI stages** (in order, every stage runs against every PR and every push to `main`; subsequent stages run only if prior stages pass):
  1. **`setup`** — `actions/setup-java@v4` (Java 17 — Android Gradle Plugin 8.x baseline); `android-actions/setup-android@v3` with the Android SDK Platform pinned to `compileSdk = 35` per Section 2 Infrastructure; `gradle/actions/setup-gradle@v3` (Gradle wrapper validation + cache setup).
  2. **`lint`** — `./gradlew lint ktlintCheck`. Android Lint runs against the merged manifest and resources; ktlint enforces Kotlin style. Both fail-fast.
  3. **`secret-scan`** — `gitleaks/gitleaks-action@v2` with `args: detect --redact --no-banner --source . --log-opts="origin/main..HEAD"` to scan the PR's diff range (full-history scan on `main` pushes). Configuration lives in `.gitleaks.toml` at the repo root and extends the default detectors with the `KEYSTORE_PASSWORD=` / `KEY_ALIAS=` / `KEY_PASSWORD=` / `.jks` / `.keystore` / `.p12` patterns from Section 3 Configuration And Secrets.
  4. **`framework-validate`** — `bash _framework/validate_framework.sh`. Fails the build if any `_context/` file contains unresolved \{\{REQUIRED:\}\}, \{\{OPTIONAL:\}\}, or \{\{EXAMPLE:\}\} markers, or if a required `_context/` file is missing. Cross-file consistency hook: this is the same validator Architect-role sessions run at task-packet close per `_context/08_active_task_packet.md` § 6 AC-001.
  5. **`arch-conformance`** — `./gradlew :*:konsistTest`. Konsist (`com.lemonappdev:konsist`) asserts the Clean Architecture layer-direction rules from `_context/04_solution_architecture.md`, the no-`android.permission.INTERNET`-imports / no-network-APIs rule from `_context/05_engineering_quality_security_and_compliance.md` § Data Protection, the `PendingIntent.FLAG_IMMUTABLE` rule from `_context/05` § OS permissions and IPC boundaries, and the no-unencrypted-`SharedPreferences`-for-sensitive-keys forward-looking rule from `_context/05` § Secret management. Konsist is preferred over ArchUnit because it is Kotlin-native (uses Kotlin compiler metadata via KSP rather than JVM reflection), has a Kotlin-DSL idiomatic rule API, and the rules file 05 specifies do not need ArchUnit's deeper graph-query features.
  6. **`unit-integration-ui-snapshot`** — `./gradlew testDebugUnitTest`. One Gradle invocation runs the entire Robolectric-hosted test suite per `_context/05_engineering_quality_security_and_compliance.md` § Engineering Quality's "all tests run under Robolectric for MVP; no `:src/androidTest/`" decision: unit, integration, contract, UI, and Roborazzi snapshot rows in the same task. Roborazzi (`io.github.takahirom.roborazzi:roborazzi`) runs *inside* Robolectric and reuses the existing test harness; Paparazzi was rejected because it runs JVM-native via LayoutLib and would re-introduce a parallel test runtime that file 05's Robolectric-only decision excludes. The 20-baseline snapshot matrix per `_context/05` § Engineering Quality (4 screens × 2 breakpoints × 2 themes + 4 notification snapshots) is recorded under `_source/<feature-module>/src/test/snapshots/`.
  7. **`coverage`** — `./gradlew koverVerify koverHtmlReport`. Kover (`org.jetbrains.kotlinx.kover` Gradle plugin) is the coverage tool, preferred over JaCoCo because it is the JetBrains-recommended Kotlin-native coverage tool with cleaner multi-module Kotlin/Android configuration. `koverVerify` enforces the per-module thresholds from `_context/05_engineering_quality_security_and_compliance.md` § Engineering Quality: `:domain` and `:core` ≥ 90 %, `:application` ≥ 80 %, no threshold on `:infrastructure` or `:presentation` (covered by integration / E2E / snapshot rows that don't lend themselves to line-coverage targets). The HTML report is uploaded as a CI artifact for the PR reviewer.
  8. **`build`** — `./gradlew assembleDebug`. Compiles the `debug` build variant per Section 1 Environments' `dev / integration` row, confirming the merge candidate produces a valid APK. `assembleRelease` runs in `release.yml`, not here.

- **Required passing checks before merge:** all eight stages above must succeed. Enforced via GitHub branch protection rule on `main`: "Require status checks to pass before merging" enabled, with `setup`, `lint`, `secret-scan`, `framework-validate`, `arch-conformance`, `unit-integration-ui-snapshot`, `coverage`, and `build` listed as required checks. "Require branches to be up to date before merging" enabled (linear merge history). "Require approvals" set to **1** with Isidro (sole human gatekeeper) as the only approver per `_context/07_delivery_plan_and_milestones.md` operating model; the approval gate is the human equivalent of `_context/08_active_task_packet.md`'s `Approved for apply` flip for Developer-role task packets.

- **Average CI duration target:** **≤ 10 minutes wall-clock** on the GitHub Actions standard `ubuntu-latest` runner. M-001 Foundation includes wiring the Gradle dependency cache + Gradle build cache (see Caching strategy below) so this target stays achievable as the test suite grows toward the file-05 coverage thresholds and the 20-baseline snapshot matrix. If a future quarter's CI duration trends past 10 minutes, the file-12 § DevOps Risks And Trade-Offs row "Empirical tuning" is the standing forum for revising this target (e.g., split the pipeline into parallel jobs, move to a larger runner, sharded test invocations); the target is not silently adjusted.

- **Caching strategy:**
  - **Gradle dependency cache** — `gradle/actions/setup-gradle@v3` (the official Gradle action) automatically caches `~/.gradle/caches/modules-2/`, keyed on `gradle/libs.versions.toml` and all `build.gradle.kts` files' hash. Restored on every CI run.
  - **Gradle build cache** — the local Gradle build cache (`~/.gradle/caches/build-cache-1/`) persisted between runs via the same `gradle/actions/setup-gradle@v3` action, keyed on the same hash. Speeds up unchanged Konsist / Kover / Robolectric invocations across PR pushes.
  - **No test-result cache in MVP** — deliberate exclusion. The Robolectric suite is small enough that re-running every PR is cheap (well under the 10-minute target with the dependency + build cache), and a stale test-result cache is a real source of "passes in CI, fails locally" failure modes that would silently undermine the merge-blocking guarantee. Re-evaluated post-MVP if the suite grows past 5 minutes wall-clock on its own.
  - **Gradle configuration cache** — enabled per Android Gradle Plugin 8.x defaults (`org.gradle.configuration-cache=true` in `gradle.properties`). Halves Gradle startup time on repeat invocations.

- **Pre-commit hook infrastructure** (resolves deferral #v from `_context/08_active_task_packet.md` § 4):
  - **Versioned hook script + Git `core.hooksPath` indirection.** The hook script lives at `_build/hooks/pre-commit` (plain bash, version-controlled, reviewable in PRs alongside any rule changes). Installation is automated by a Gradle task `installGitHooks` that runs `git config core.hooksPath _build/hooks` on first invocation; the task is wired as a dependency of `preBuild` via `tasks.named("preBuild") { dependsOn("installGitHooks") }` so any developer (or Claude Code agent) running `./gradlew assembleDebug` for the first time after `git clone` automatically installs the hook. Idempotent — the task no-ops if `git config core.hooksPath` already returns `_build/hooks`.
  - **What the pre-commit hook runs** (fast path; full CI re-runs everything in `.github/workflows/ci.yml`):
    - `gitleaks protect --staged --redact --no-banner` — secret scan against the staged diff before commit; rejects the commit on any detector hit.
    - `bash _framework/validate_framework.sh` — framework validator; rejects the commit if any `_context/` file regresses on placeholder markers or required-file presence.
    - `./gradlew ktlintCheck --quiet` — Kotlin style check on staged files only (via the ktlint Gradle plugin's `--include` argument scoped to `git diff --name-only --cached`).
    - **Not** included in the pre-commit hook: Android Lint, Konsist, the Robolectric suite, Kover, or `assembleDebug`. Those run in CI; including them locally would push pre-commit wall-clock past the ~5-second budget that keeps the hook unannoying.
  - **Lefthook and Husky were rejected** because both add a binary dependency (Go binary or Node runtime + `package.json`) to an otherwise pure-Kotlin/Gradle project; the plain-bash script + `core.hooksPath` indirection has zero external dependencies and equivalent expressive power for the four checks above.

- **Secret-scanner vendor:** **gitleaks** (`zricethezav/gitleaks` upstream; pinned at a specific release tag in both the pre-commit hook and the `.github/workflows/ci.yml` `gitleaks/gitleaks-action@v2` invocation). Configuration in `.gitleaks.toml` at the repo root, extending the default detector pack with the project-specific patterns enumerated in Section 3 (`KEYSTORE_PASSWORD=`, `KEY_PASSWORD=`, `KEY_ALIAS=`, plus the `.jks` / `.keystore` / `.p12` filename rules). truffleHog and detect-secrets were considered and rejected: truffleHog is excellent for long-history deep scans but heavier for the pre-commit fast path; detect-secrets has a smaller adoption footprint in the Android/Kotlin ecosystem and a less mature GitHub Actions integration.

## Continuous Delivery And Deployment

- **Branch strategy:** **Trunk-based with short-lived feature branches.** All work happens on branches off `main`, opened as pull requests, merged back into `main` through the merge gate defined in Section 4 Continuous Integration. No long-lived `develop`, `release`, or `hotfix` branches — GitFlow ceremony does not earn its cost for a solo-developer Android project with a manual Play Console promote step. Feature branches are deleted after merge. Tags are used to mark released `versionCode`s (`v1.0.0`, `v1.0.1`, etc.) but do not branch the working tree.

- **Merge contract:** Re-affirms and operationalizes the merge-blocking gate already pinned in Section 4 Continuous Integration:
  - **1 approval required** on every PR targeting `main`, with Isidro as the sole approver per `_context/07_delivery_plan_and_milestones.md` operating model (Isidro is the human gatekeeper; QA / Security / Developer roles are filled by Claude Code agents and do not register as GitHub approvals).
  - **All 8 CI status checks** (`setup`, `lint`, `secret-scan`, `framework-validate`, `arch-conformance`, `unit-integration-ui-snapshot`, `coverage`, `build` per Section 4) listed as **required status checks** in the `main` branch protection rule.
  - **"Require branches to be up to date before merging" enabled** — every merge candidate is rebased onto the current `main` tip before the CI re-runs, preserving a linear merge history.
  - **"Allow force pushes" disabled** on `main`. **Direct push to `main` disabled** ("Restrict who can push to matching branches" with an empty allow-list).
  - **For Developer-role task packets, the `_context/08_active_task_packet.md` `Approved for apply` flip is the *substantive* gate** (it authorizes the Developer role to mutate `_source/`). The GitHub branch-protection approval is the *mechanical* enforcement on the resulting PR.

- **Deployment trigger:** **Manual promote via the Google Play Console web UI.** No auto-publish from CI. Merge to `main` runs the eight-stage CI pipeline (per Section 4); it does **not** upload anything to Play Console. When Isidro decides a release is ready, the manual sequence is:
  1. Trigger `.github/workflows/release.yml` from the GitHub Actions UI (`workflow_dispatch`). The workflow runs the same eight stages plus an additional `assembleRelease` step that signs the AAB with the upload key (per Section 3 Configuration And Secrets — Secrets row) and uploads the signed AAB as a workflow artifact.
  2. Download the signed AAB from the workflow artifact.
  3. Manually upload the AAB to **Play Console → Internal Testing track**; install on a personal device via the Play Internal Testing tester list (the `internal` environment from Section 1 Environments); run the M-004 manual Doze test on real hardware per `_context/07_delivery_plan_and_milestones.md` M-004 Done When.
  4. If the M-004 validation passes, manually promote the same release from Internal Testing → Production via Play Console's "Promote release" affordance.

  Two human-driven steps total (release.yml run, Play Console upload-then-promote). Both gate on Isidro's deliberate action; neither happens automatically on merge.

- **Deployment topology** (vocabulary repurposed from the framework template's server-oriented "rolling / blue-green / canary / immutable / in-place" enumeration, which does not fit on-device-only Android distribution):
  - **Each release is immutable.** Identified by a monotonically incremented `versionCode` integer plus a human-readable `versionName` string (semver-ish, e.g., `1.0.0`, `1.0.1`, `1.1.0`). Once a `versionCode` has been promoted to the Play Console Production track, the underlying AAB cannot be re-issued under the same `versionCode`; any change — including a one-character bug fix — requires a new `versionCode`.
  - **No staged rollout.** Production-track releases ship to **100 % of eligible users immediately** when promoted from Internal Testing → Production. The Play Console staged-rollout affordance is **not used** in MVP. **Trade-off:** the halt-rollout safety net (Play Console's "Halt rollout" button, available only when a staged release is mid-progression) is *not available* as a mitigation; once a bad release reaches the Production track, every eligible user can receive it, and the sole recovery path is fix-forward (see Rollback mechanism below). This is an explicit solo-developer simplification — the staged-rollout discipline of 48-hour observation windows between percentage stages competes with the gatekeeper's limited review-and-approval bandwidth, and the M-004 manual Doze test on the `internal` Play track is treated as the pre-production gate that staged rollout would otherwise be. The trade-off is recorded as a Section 10 DevOps Risks And Trade-Offs row so it stays visible.
  - **Internal Testing track delivery is immediate 100 %.** Only Isidro's devices are enrolled in the Internal Testing tester list; staged rollout adds no value at this tier regardless of the production-track policy.

- **Database migration strategy:** **Forward-only Room migrations with explicit `Migration` objects per schema version increment, no destructive fallback.** Concrete rules:
  - Every change to a `@Entity` class that alters the SQLite schema **increments** the Room `@Database(version = N)` integer.
  - Every increment ships with a corresponding `androidx.room.migration.Migration(fromVersion, toVersion)` object in the database module, registered via `Room.databaseBuilder(...).addMigrations(...)` at app-startup wiring in `:app::di::AppModule`.
  - **`fallbackToDestructiveMigration` is *not* used.** If a migration fails at startup, the app crashes rather than silently dropping every Medication, Dose, and Confirmation — which would violate `_context/05_engineering_quality_security_and_compliance.md` § Data Protection sub-rule (c) "Patient-controlled hard deletion" (deletion is *Patient-controlled*, not migration-triggered). A failed-migration crash is treated as a P0 bug requiring fix-forward.
  - **Schema-history file required.** `_source/<datastore-module>/schemas/com.nemopill.<database-class-name>/<version>.json` is the Room annotation-processor output committed to the repo per Room migration-testing conventions (`@Database` `exportSchema = true` plus `room.schemaLocation` Gradle argument). The Konsist architecture-conformance suite from Section 4 asserts that every `@Database` version bump has both a committed schema JSON for the new version and a registered `Migration` object from the previous version, failing the merge if either is missing.
  - **No downtime window.** Migrations run on each Patient's device on first launch after the upgrade. Typical migration wall-clock for the NemoPill data volume is sub-second; the Patient experiences a one-time slower first-launch, subsequent launches are normal.

- **Rollback mechanism:** **Fix-forward only.** The halt-rollout path traditionally available in Play Console is **not applicable** to NemoPill releases because the "Deployment topology" decision above commits to no staged rollout — there is no in-progress staged rollout to halt. The single recovery path for any released bug is:
  1. **Bump the `versionCode`** (e.g., `42` → `43`); leave the `versionName` reflecting whether this is a patch (`1.0.1` → `1.0.2`) or a hotfix (`1.0.1-hotfix.1`).
  2. **Fix the bug on a branch off `main`**, including a regression test row in whichever Section 4 CI stage covers the broken behavior (unit / integration / contract / UI / Roborazzi snapshot / Konsist).
  3. **Run the full eight-stage CI pipeline** via the standard PR merge path; do not bypass any required check, including under perceived time pressure.
  4. **Promote through Internal Testing → Production** using the standard manual sequence (Deployment trigger above), with the M-004 manual Doze test re-run on the `internal` track before promoting to Production.
  5. **Target wall-clock times** depend on bug severity, not on a committed SLA:
     - **P0** (data loss, app uncrashable, security exposure): same calendar day if Isidro is available; sole gatekeeper means no on-call backup; the M-004 hardware test cannot be skipped because the M-004 hardware test is what catches Doze / boot-survival regressions that would otherwise compound the original bug.
     - **P1** (significant feature broken, major UX regression): within the work-week of the gatekeeper.
     - **P2** (minor regression, cosmetic, low-impact): folded into the next normal release.
  6. **For broken Room migrations specifically** — the worst case, because every Patient who launches the new version hard-crashes on first-launch migration, and existing Patient data is intact on disk but inaccessible to the app. Mitigation discipline: fix-forward as above *plus* wrap the failing migration in a defensive `try/catch` that surfaces a one-screen "Update required — see Play Store for the fix" message rather than a raw crash dialog. The Konsist schema-JSON + Migration-object invariants from the Database migration strategy bullet above catch most migration bugs in CI before they reach Patient devices; the defensive wrapper is the second line of defense.

- **Rollback rehearsal cadence:** **Not applicable for MVP.** Rationale: rollback rehearsal is a server-operations discipline that does not translate to a solo-developer Android app — there is no staging environment to halt a rollout against (no staged rollout exists per the Deployment topology bullet above), no on-call rotation to drill, and the fix-forward path's mechanical steps (bump `versionCode`, fix on a branch, run CI, promote through Internal Testing → Production) are the same steps as any normal release and so are exercised on every release. **Follow-up condition:** if a post-MVP fix-forward release ever happens in anger, that incident is the rehearsal and this row is revisited in a follow-up task packet — for example, to commit to a quarterly tabletop walkthrough of the Room-migration worst-case scenario, or to introduce a staged-rollout discipline that would reopen the halt-rollout option in Section "Deployment topology" above.

## Release Process

- **Release cadence:**
  - **MVP (M-001 → M-006): per-milestone.** Internal Testing track releases happen at the close of each `M-###` that ships user-visible behavior — predominantly an M-004 hardware-validation cycle for each Foundation increment that touches the BR-### scheduling pipeline. Production-track releases happen **only at M-006** (the MVP public release) per `_context/07_delivery_plan_and_milestones.md`. Intermediate M-### releases stop at the Internal Testing track.
  - **Post-MVP: release-when-worth-releasing.** No calendar cadence. A release happens when a bug fix or feature is ready *and* the gatekeeper has bandwidth to run the M-004 hardware Doze test on the `internal` track (per Section 5 Continuous Delivery And Deployment, Deployment trigger). This matches the solo-developer operating model from `_context/07_delivery_plan_and_milestones.md` — bandwidth constraints are review-and-approval bandwidth, not implementation throughput, and a release is the canonical bandwidth consumer.

- **Release approver:** **Isidro** (sole human gatekeeper per `_context/07_delivery_plan_and_milestones.md` operating model and the `_context/08_active_task_packet.md` `Approved for apply` flip discipline). The gatekeeper's approval is the same human gate that flips `Task status: Approved for apply` for Developer-role task packets; the Production-track promote is the operational analogue for release work.

If a P0 fix-forward needs to ship and the gatekeeper will be unreachable for the post-release observation period, **hold the release until the gatekeeper is back** — the trade-off between letting a P0 bug sit one more day versus releasing a fix without anyone available to halt-or-fix-forward the *replacement* if the fix has its own bug favors holding, given the Section 5 "no staged rollout, fix-forward only" recovery model.

## Monitoring And On-Call

- **Primary monitoring tool:** **Not applicable.** NemoPill is on-device-only with no remote service (per `_context/04_solution_architecture.md` § Solution Architecture and Section 2 Infrastructure of this file), no SLO to track, and no metrics backend to query. `_context/05_engineering_quality_security_and_compliance.md` § Engineering Quality already settled "metrics / tracing / alerting `Not applicable`"; this bullet mirrors that prior decision rather than re-deciding it.

- **Error tracking tool** (resolves deferral #10 from `_context/08_active_task_packet.md` § 4 — crash-reporting vendor with the file-05 no-`INTERNET` rule as a binding side-constraint and the file-06 "no Patient data in failure evidence" hard rule as a second binding side-constraint):
  - **Decision:** **Google Play Console "Android Vitals" crash reporting only.** No third-party crash reporter (no Firebase Crashlytics, no Sentry, no Bugsnag, no Rollbar). No custom on-device crash log file in MVP.
  - **Why this satisfies both binding constraints:**
    - **No `android.permission.INTERNET` required.** Android Vitals reports are collected and uploaded by Google Play Services running on the user's device — Play Services has its own network permission set, separate from the NemoPill app process. NemoPill itself does not open a socket, perform a DNS lookup, or import a network API; the Section 4 Konsist arch-conformance suite continues to assert that no class in any module imports `java.net.*`, `okhttp3.*`, `retrofit2.*`, or any analogous network API.
    - **Anonymized stack-trace-only payloads.** Play Console Android Vitals reports the stack trace plus device metadata (Android version, device model, free memory at crash time, similar). It does **not** report app-private database contents, `SharedPreferences` values, `Intent` extras, notification payloads, or any other in-app state.
  - **Required defensive discipline** (to honor the file-06 "no Patient data in failure evidence" rule):
    - **No Patient-input-derived value in any `throw` statement's message argument.** Exception messages must be Patient-data-free constants or contain only values derived from build-time constants / non-Patient data (e.g., `throw IllegalStateException("Scheduling worker enqueued without medicationId")` is fine; `throw IllegalArgumentException("Invalid medication: ${medication.name}")` is **not**, because `medication.name` is Patient input).
    - **No Patient-data class's `toString()` implementation surfaces Patient-input values.** The `data class` auto-generated `toString()` for `Medication`, `Dose`, `Confirmation`, and any value object holding Patient input (`Strength`, `Form`, `TimeOfDay`, `DateRange`, `Frequency`, `DoseQuantity` per `_context/05_engineering_quality_security_and_compliance.md` § Input validation) must be overridden to emit a redacted form (e.g., `Medication(id=<uuid>, name=<redacted>)`).
    - **Enforcement point:** a Konsist rule in the Section 4 `arch-conformance` CI stage asserts (i) no `throw` statement uses Kotlin string templates (`${...}`) or `String.format` / `StringBuilder.append` calls inside its message argument, and (ii) every `data class` in the Domain layer (per `_context/04_solution_architecture.md` six-module Gradle layout — `:domain` and `:core`) overrides `toString()` to a redacted form. Both rules are part of the existing Section 4 arch-conformance suite; no new CI stage is introduced.
  - **Patient opt-out path:** Patients can disable Android Vitals collection at the OS level via Google Play Services → Usage & diagnostics → "Help improve Android" toggle. The setting is per-device and applies to all apps; NemoPill does not add a separate in-app opt-out because the OS-level control is the canonical surface and adding a parallel in-app toggle would duplicate state and create reconciliation bugs. This is mentioned in the bilingual Privacy Policy (per Section 6 Privacy Policy bullet) as part of the "data we share with Google" disclosure.
  - **Revisit condition:** if post-MVP shows Android Vitals is missing a meaningful slice of crashes (e.g., devices in markets without Google Play Services, or a class of `Result.Err.Unexpected` outcomes that don't crash but should be visible), this bullet is revisited toward the (d) combination (Android Vitals + on-device crash log file with Patient-driven export) considered and deferred in this session.

- **Dashboards required at launch:** **One pre-built dashboard, zero custom dashboards.** The only dashboard in scope is the Google Play Console "Android Vitals → Crashes & ANRs" dashboard, pre-built by Google and surfaced at `play.google.com/console/.../app/<app-id>/vitals/crashes`. The link is recorded in `_build/play-store-state.md` (per Section 6 Play Console state tracking bullet) when that file is initialized at M-006 pre-launch, so a future gatekeeper or audit reader can find it from the repo without logging into Play Console. **No custom dashboards** — no project-built Grafana board, no Looker / Metabase / Superset surface, no in-app debug dashboard. The Section 4 `coverage` CI stage's HTML report is uploaded as a PR artifact (per Section 4) and is the closest analogue to an internal dashboard for the project, but it is per-PR, not standing.

- **On-call rotation tool:** **Not applicable.** Solo developer per `_context/07_delivery_plan_and_milestones.md` operating model; no rotation to schedule, no PagerDuty / Opsgenie / VictorOps tooling to wire up, no escalation chain to define. Isidro is the only responder for any operational issue.

- **On-call expectations:** **Not applicable as an SLA.** The fix-forward severity targets in Section 5 Continuous Delivery And Deployment Rollback mechanism — P0 same-day if gatekeeper available, P1 within the work-week, P2 next normal release — plus the pre-OOO 24-hour rule in Section 6 Release Process Freeze windows are the only response-time commitments the project makes, and they are explicitly **operating norms for a solo gatekeeper, not SLAs to external stakeholders**. There is no escalation chain (no second-on-call, no manager, no rotation backup), no after-hours expectation, and no committed Patient-facing response time. The bilingual Privacy Policy (per Section 6) does not promise any response time for Patient-reported issues; if a future product surface introduces a support email or similar Patient-feedback channel, this bullet is revisited.

## Backup, Recovery, And Disaster Plan

- **Backup scope:** **Not applicable for Patient data.** No Patient data is backed up by design. Three composing rules already settled in earlier `_context/` files and Section 3 of this file make backups unavailable:
  - `_context/05_engineering_quality_security_and_compliance.md` § Data Protection sub-rule (b) — **no off-device transmission** — and the absent `android.permission.INTERNET` per Section 2 Infrastructure of this file. Any cloud or remote backup destination crosses this commitment by definition.
  - Section 3 Configuration And Secrets, Build-time configuration row — **`android:allowBackup="false"`** plus the Android 12+ `dataExtractionRules` reference that excludes all app data from both `<cloud-backup>` and `<device-transfer>`. The Android Auto Backup and Android device-to-device transfer surfaces are explicitly closed.
  - `_context/06_integrations_data_flow_and_boundaries.md` § Data Ownership And Retention — **uninstall = total deletion**, plus `_context/05_engineering_quality_security_and_compliance.md` § Data Protection sub-rule (c) — **Patient-controlled hard deletion is irreversible** ("there is no recovery, no backup, no undo").

  The Room database lives only in the app's private internal storage on the Patient's device. If the Patient loses the device, replaces the device, factory-resets, or uninstalls NemoPill, the data is gone — this is a deliberate product-level trade-off rooted in `_context/01_product_vision_and_scope.md`'s "medication data must not be transmitted or shared without explicit Patient consent" commitment, not a backup-feature gap.

- **Backup cadence:** **Not applicable.** Flows directly from Backup scope above — there is no backup operation to schedule. The bilingual Privacy Policy authored at M-006 (per Section 6 Privacy Policy bullet) explicitly tells Patients that NemoPill does not back up their data and that uninstalling the app erases everything, so the absence of a backup cadence is disclosed Patient-facing, not hidden.

- **Restore drill cadence:** **Not applicable.** Nothing to restore from on the Patient-data side. Recovery for project-side asset loss (upload signing key, source repository, `_build/play-store-state.md`) is exercised at the moment of need — see the Project asset recovery bullet below — rather than rehearsed on a calendar; the volumes are too small and the recovery procedures too rare for periodic drills to earn their cost.

- **Disaster recovery plan reference:** **Not applicable as a standalone plan document.** No project-controlled remote service to fail over, no SLO to honor, no team to coordinate, no committed Patient-facing uptime. The only DR-like scenarios are (i) upload signing key loss (covered in the Project asset recovery bullet below), (ii) source-repository host outage (GitHub is the project's only repo host; if GitHub is unreachable, releases simply pause until it returns — adding a failover destination would not earn its cost at this scale), and (iii) Play Console account compromise (recovery is via Google account recovery flows, not via a project-managed plan). If a post-MVP scenario surfaces a need for a standalone DR document, it is added then as `_build/disaster-recovery.md` and this bullet is revised.

- **Project asset recovery** (new bullet outside the framework template's vocabulary; captures the only meaningful recovery story for NemoPill — the project-side assets, not Patient data):

  - **Upload signing key** (`~/.android/nemopill-upload-keystore.jks` + its keystore password, key alias, and key password, per Section 3 Configuration And Secrets — Secrets row). **The single irreplaceable project asset.** Loss means a multi-week Google-mediated "upload key recovery" ceremony to ship any new release under the same Play Store app ID; compromise means an attacker could potentially sign and distribute a replacement APK under the project's identity, hijacking every Patient install.

    - **Required local-backup discipline:** keep at least one copy of the keystore file and its credentials on an **offline, encrypted external medium** (e.g., a hardware-encrypted USB drive stored physically separate from the development laptop, an encrypted Time Machine backup on a local-only disk, or a hardware security key with file storage). The medium must be **disconnected from the internet whenever NemoPill-related data resides on it**, mirroring the no-off-device-transmission spirit of `_context/05_engineering_quality_security_and_compliance.md` § Data Protection sub-rule (b).

    - **Forbidden backup paths (explicit):** **no cloud storage** of any kind — no iCloud Drive, Google Drive, Dropbox, OneDrive, Box, or equivalent; **no CI runner secret** mirror (Section 3 Configuration And Secrets — Secrets row already settled CI does not see the keystore); **no shared drive** (NFS, SMB, Synology, etc.) accessible over a network; **no git submodule** or any other repo-resident copy (the secret-scanner from Section 4 Continuous Integration would reject this anyway, but the rule is stated for clarity); **no off-device backup that contradicts file 05 § Data Protection sub-rule (b)'s spirit**, even if the backup destination is encrypted at rest — the rule is about the *path*, not the resting state.

    - **Recovery path on key loss:** Google Play Console → App integrity → "Use a different upload key" flow. Google verifies the project's control of the Play Console account through their own recovery mechanism (typically requiring access to the Google account that owns the Console, plus a multi-day cool-down for the key upgrade to take effect on the user-facing app signing key). Expect a multi-week wait between key-loss event and the next viable release. The mechanical step-by-step procedure is captured in the eventual Section 9 Operational Runbooks "Upload key recovery" runbook when Section 9 is drafted.

  - **Source repository.** The canonical copy is the GitHub remote at the project's `origin`; redundancy is each developer machine's local clone (in MVP, Isidro's primary laptop, optionally a secondary machine). Recovery on GitHub-side data loss is the standard `git push --mirror` from any complete local clone to a new remote; recovery on local-machine loss is `git clone` from GitHub. **No special backup discipline required** beyond standard developer hygiene — do not rely on a single local clone as the only copy, and keep the local clone on a machine that participates in your normal personal-data backup regime (e.g., Time Machine on the development laptop, which is a non-NemoPill backup surface and is not constrained by file 05 because the *Patient data* is not on this machine; the *source code* is, and source code is not Patient data).

  - **`_build/play-store-state.md` and Play Console state.** Per Section 6, this file is introduced at M-006 and is the version-controlled mirror of Play Console state. **Recovery on Play Console state loss** (e.g., an accidental edit in Console UI that wipes a field; a Console-side bug; an account-recovery event that resets configuration): the committed `_build/play-store-state.md` in the source repository is the source-of-truth for re-creating the listing, country list, store-listing text, screenshot references, and `versionCode` ledger. **Recovery on Play Console *account* loss** (e.g., the Google account that owns the Play Console is compromised or locked out): recovery is via Google account recovery flows — outside the project's control surface; the project does not commit to a recovery time. If post-MVP introduces a second Play Console admin (currently Isidro is the sole admin), this row is revisited.

## Operational Runbooks

NemoPill registers four operational runbooks in this section. Each runbook is a Markdown file under `_build/runbooks/`. None of the four runbook files exist at file-12 contextualization time — the table below pins names, paths, owners, and authoring milestones; the files themselves are authored at the milestone listed in each row's `Authoring status` column.

**Standard runbook structure (recommended, not Konsist-enforced):** each `_build/runbooks/<name>.md` follows a shared scaffold so the future Architect-role author at M-006 starts from a known shape rather than reinventing per runbook —

- `## When to use` — the conditions that trigger the runbook.
- `## Prerequisites` — what state the gatekeeper must confirm before starting (e.g., access to the Google Play Console, current `versionCode` known, branch checked out).
- `## Procedure (numbered steps)` — the mechanical step-by-step. Every step is concrete enough that the gatekeeper, or a future Claude Code agent executing the runbook, does not have to make an unanchored judgment call.
- `## Verification` — how to confirm the runbook succeeded; the executable check that the gatekeeper runs after the last procedural step.
- `## Rollback / abort` — the recovery path if a step fails mid-procedure.
- `## Open-loop questions for the gatekeeper` — explicit "stop and ask Isidro" points; intentional gates rather than hidden assumptions.

This scaffold is a recommendation pinned here; it is **not** enforced by Konsist or any CI check (runbook files live outside `_source/` and outside the framework validator's scope). The discipline is procedural.

## DevOps Risks And Trade-Offs

This section is the canonical synthesis surface for trade-offs accepted across Sections 1–9 of this file and forward from earlier `_context/` files. A future reader — `_context/09_decision_log.md` (T-003) ADR backfill, `_context/13_threat_model_and_data_classification.md` (T-005) threat-model contextualization, post-MVP scope-revision task packets, or external auditor — should be able to consume this section without re-reading Sections 1–9 and still understand what was deliberately *not* done in MVP and what is queued for later.

- **Accepted operational gaps for MVP** (decisions to live with the gap; revisited only if the trigger condition specified in each row materializes):

  - **No staged rollout; halt-rollout safety net unavailable; fix-forward is the sole recovery path.** The single most important risk accepted in MVP. Production-track releases ship to 100 % of eligible users immediately when promoted from Internal Testing → Production (Section 5 Continuous Delivery And Deployment Deployment topology bullet). Once a bad release reaches Production, Play Console's "Halt rollout" affordance is *not available* as a mitigation — the staged-rollout discipline of 48-hour observation windows between percentage stages was explicitly rejected as too expensive for the solo-developer gatekeeper bandwidth, and the M-004 manual Doze test on the `internal` Play track is treated as the pre-production gate that staged rollout would otherwise be. **Consequence:** the sole recovery for any released bug is the Fix-forward runbook (Section 9), with the P0 / P1 / P2 wall-clock framing from Section 5. **Trigger to revisit:** a P0 fix-forward release in anger that would have benefited from halt-rollout — that incident itself reopens the staged-rollout discipline in a follow-up task packet that revises the Section 5 Deployment topology bullet.

  - **Solo gatekeeper, no on-call rotation, no committed SLA.** Isidro is the only responder; there is no second-on-call, no manager, no rotation backup; the fix-forward severity targets (Section 5 Rollback mechanism: P0 same-day if gatekeeper available, P1 within the work-week, P2 next normal release) plus the pre-OOO 24-hour rule (Section 6 Release Process Freeze windows) are operating norms for the solo gatekeeper, **explicitly not SLAs to external stakeholders**. The bilingual Privacy Policy (Section 6) does not promise any response time for Patient-reported issues. **Trigger to revisit:** addition of a second human collaborator (formally — adding a Play Console admin counts; informally — having Isidro mention "I asked Maria to look at it" does not).

  - **No backup of Patient data; uninstall = irreversible total loss.** Three composing rules already settled (`_context/05_engineering_quality_security_and_compliance.md` § Data Protection sub-rule (b) "no off-device transmission" + Section 3 Configuration And Secrets `android:allowBackup="false"` and Android 12+ `dataExtractionRules` exclusion + `_context/06_integrations_data_flow_and_boundaries.md` § Data Ownership And Retention "uninstall = total deletion") make Patient-data backups unavailable by design (Section 8 Backup, Recovery, And Disaster Plan). The Patient losing their device, factory-resetting, or uninstalling NemoPill permanently destroys their Medication, Dose, and Confirmation history. **This is disclosed Patient-facing in the bilingual Privacy Policy authored at M-006** (Section 6 Privacy Policy bullet) — it is not a hidden gap. **Trigger to revisit:** a product-scope revision that introduces Patient-driven export or sync, which itself requires revising `_context/01_product_vision_and_scope.md`, `_context/05_engineering_quality_security_and_compliance.md` § Data Protection, and Section 8 of this file.

  - **No analytics SDK; no telemetry; no off-device crash reporting beyond Play Console Android Vitals.** Reaffirms `_context/05_engineering_quality_security_and_compliance.md` § Engineering Quality "No analytics SDK in MVP." Production-crash visibility is limited to the Play Console "Android Vitals → Crashes & ANRs" dashboard (Section 7 Monitoring And On-Call). The gatekeeper accepts limited Production-crash visibility as the trade-off for the no-`android.permission.INTERNET` commitment and the no-Patient-data-in-failure-evidence discipline. **Trigger to revisit:** Android Vitals missing a meaningful slice of crashes (e.g., devices in markets without Google Play Services).

  - **Manual Play Console upload + manual promote; no CI-automated Play upload.** Every release requires Isidro to trigger `release.yml`, download the signed AAB, manually upload to Play Console Internal Testing, run the M-004 hardware Doze test, then manually promote Internal Testing → Production (Section 3 Configuration And Secrets — Secrets row option (a); Section 5 Continuous Delivery And Deployment Deployment trigger). **Trade-off:** the human bottleneck *is* the gatekeeper-bandwidth model from `_context/07_delivery_plan_and_milestones.md` and keeps the upload-key exposure surface to a single laptop. Manual release toil is the cost. **Trigger to revisit:** release cadence (Section 6 Release Process) exceeding the gatekeeper's bandwidth post-MVP.

  - **Pre-commit hook fast-path checks only.** The pre-commit hook (Section 4 Continuous Integration — Pre-commit hook infrastructure bullet) runs only gitleaks + framework validator + `ktlintCheck` scoped to staged files; Lint / Konsist / Kover / Robolectric / `assembleDebug` run only in CI, not pre-commit. **Trade-off:** unannoying commits (~5-second budget) at the cost of slower feedback on a Konsist rule violation or a Kover coverage regression (visible at CI time, ~minutes after push, rather than at commit time). **Acceptable because** CI runs anyway on every PR and Konsist / Kover failures block merge.

  - **No test-result cache.** Section 4 Continuous Integration deliberately excludes test-result caching even though Gradle dependency cache and Gradle build cache are wired. **Trade-off:** marginally slower CI on identical re-runs in exchange for eliminating an entire class of "passes in CI, fails locally" failure modes from stale cache.

  - **No automated CVE scanning of Gradle dependencies.** Reaffirms `_context/05_engineering_quality_security_and_compliance.md` § Dependency and supply chain "Manual review." Every new direct dependency or meaningful transitive update is reviewed by Isidro at commit time, with a one-line justification in the commit message. **Trade-off:** slower CVE detection (no `OWASP dependency-check`, no Snyk, no Dependabot security alerts beyond GitHub's defaults) in exchange for tooling simplicity. **Acceptable because** the dependency graph is small at MVP and the manual-review baseline catches CVE-adjacent decisions at the moment of introduction. **Trigger to revisit:** dependency graph growth past the gatekeeper's manual-review capacity, or activation of a regulated standard in the file-05 Compliance Applicability Matrix that mandates automated scanning.
