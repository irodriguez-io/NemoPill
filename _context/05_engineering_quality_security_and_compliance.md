# Engineering Quality, Security, And Compliance

## How To Fill This File

- Define the minimum engineering bar the coding agent must satisfy on every task.
- Keep the compliance matrix accurate. If a standard is not relevant, say so explicitly.
- The Compliance role defined in `14_agent_roles_and_handoffs.md` is inactive unless this file marks at least one standard as `Applicable` or `Planned`.

## TDD Workflow

1. **Tests are proposed in Gherkin/BDD form inside the active task packet.** When the Developer role enters Proposal Mode, it writes a *Tests To Add First* subsection in `_context/08_active_task_packet.md` that lists the scenarios the task must cover, by reference to the BDD scenario IDs already in `_context/03_user_experience_and_use_cases.md` (e.g., `BDD-CON-02`, `BDD-SCH-04`). Where the task introduces behavior not already covered by a file-03 scenario, additional Given/When/Then scenarios are written inline in the packet using the same vocabulary as file 03 and the Ubiquitous Language from `_context/01_product_vision_and_scope.md`. The proposal does **not** include Kotlin test stubs, file paths, or method names — those are an Apply Mode concern. The proposal stays at the behavioral level: which business rules (`BR-001`–`BR-012`) and aggregate invariants will be protected, which Adherence-relevant edges (`BR-008` / `BR-010` / `BR-011` / `BR-012`) will be exercised, and which use cases (`UC-001`–`UC-009`) will be invoked. The `Tests First` section of every Proposal Mode chat response (per the response structure in `CLAUDE.md`) mirrors the same scenario list — keep them identical so the packet and the reply cannot drift.

2. **Two-pass review with QA before the human approves.** The Developer's Proposal Mode response yields to a QA-role turn that validates the proposed scenario list against (a) the 21 BDD scenarios in `_context/03_user_experience_and_use_cases.md`, (b) the business rules `BR-001` through `BR-012` in `_context/02_domain_model_and_business_rules.md`, and (c) the relevant aggregate invariants and domain events also defined in file 02. The QA turn renders a verdict — `Coverage Sufficient` or `Coverage Insufficient` with a specific gap list (for example, "missing scenario for `BR-011` retroactive window expiry at exactly `scheduledAt + 24h`"). If QA returns `Coverage Insufficient`, the Developer revises the proposal and QA re-reviews; this loop runs until QA returns `Coverage Sufficient`. Only then does the human review the QA verdict and the revised packet, and only the human may set `Task status: Approved for apply` in `_context/08_active_task_packet.md` per the Apply Mode rules in `CLAUDE.md`. QA cannot self-approve; the Developer cannot bypass QA.

3. **Four engineering gates must all pass before the task is considered complete.**
    1. **Tests first, red-then-green.** Every scenario proposed in step 1 has been turned into one or more Kotlin tests that were observed to fail before the implementation was written, and now pass. The red-then-green discipline is checked by inspecting commit ordering or, where commits are squashed, by the Developer's explicit verification in the response's `Tests And Verification` section.
    2. **Full project test suite passes.** `./gradlew test connectedAndroidTest` (or the equivalent gate locked in the `Quality Gates` section below and finalized in `_context/12_environments_and_devops.md`) exits zero across every Gradle module.
    3. **Lint and static analysis pass.** ktlint and detekt (versions pinned in file 12) exit zero.
    4. **Domain purity rule holds.** Every feature module's `:domain` package compiles and runs its tests with only `kotlin-stdlib`, `junit`, and `truth` on its classpath, verified by a build-time check using Konsist or ArchUnit (tool choice deferred to file 12). If any of these four gates fails, the task is not complete and Apply Mode work continues until they do. These are *engineering* verification gates only — the framework obligations to update `_context/` where documented truth changed, to append an ADR to `_context/09_decision_log.md` when a durable architecture / policy / integration / security / delivery decision was introduced, and to append a structured entry to `_context/handoff_log.md` remain non-negotiable per `CLAUDE.md` and are checked separately at the end of every role turn.

### Notes on the workflow boundary with adapter code

- For pure-Kotlin Domain code and Application use cases, the workflow above is **strict tests-first**: the failing JUnit test exists in source control (or has at minimum been run red locally) before the production code is written.
- For adapter code that depends on the Android SDK (`AlarmManagerSchedulerAdapter`, `BootCompletedReceiver`, `TimezoneChangedReceiver`, the `NotificationManagerCompat`-backed `NotificationPort` adapter, Room DAOs and `TypeConverter`s), the workflow is **pragmatic tests-first**: an initial implementation sketch may exist before its Robolectric / in-memory-Room tests are written, but the adapter is **not considered done** until those tests exist, are green, and have been reviewed under the same step-2 QA coverage gate as the Domain and Application tests. The boundary exists because Robolectric / Room scaffolding is verbose to set up against an empty file, and a short implementation sketch frequently surfaces testability issues that pure speculation does not. The strict-vs-pragmatic split is a workflow choice, not a coverage exemption: adapter-level coverage of the business rules the adapter implements (`BR-004` exactly-once Reminder firing, `BR-007` archive-delete-future-pending atomicity, `BR-012` timezone-change regeneration) is still required by step 1 and validated by QA in step 2.

- **Coverage tool choice** (JaCoCo vs. Kover) and the **architecture-conformance tool** (Konsist vs. ArchUnit) — file 12.
- **Snapshot tool choice** (Roborazzi vs. Paparazzi) — file 11 / file 12.
- The **`Out Of Scope For This Task`** verdict on the Security gate exists so file 05 doesn't force every routine task (e.g., a Compose-only UI tweak with no permission, no I/O, no dependency change) to drag the Security role through a full review. ## Observability Expectations

- **Required logs or events:** Minimal logging. Only the `Result.Err.Unexpected` branch of the two-tier error policy from `_context/04_solution_architecture.md` is logged, via `Log.e` (or `Log.w` for recoverable adapter-level surprises that nonetheless propagate up as `Unexpected`). Tags follow the convention `NemoPill.<module>.<class>` so logcat filtering is straightforward. Domain events — `MedicationCreated`, `DoseScheduleReplaced`, `MedicationArchived`, `DoseMaterialized`, `ReminderFired`, `DoseConfirmed`, `DoseMissed`, `DeviceTimezoneChanged` — are **not** separately logged: they are already observable through the `DomainEventPublisher` (`InProcessEventBus` backed by `MutableSharedFlow`), and a developer who wants to introspect a flow can attach a temporary `Log`-only subscriber during debugging. Expected business-rule failures (`BR-002` invalid date range, `BR-003` zero/negative quantity, `BR-011` retroactive-window expired, etc.) are *never* logged — they are returned as `Result.Err` to the Application boundary and surfaced to the Patient through UX copy, not noise in logcat.

- **Metrics or service indicators:** `Not applicable`. NemoPill has no service surface and no metrics backend — there is no latency, throughput, error rate, or queue depth to measure. The product-level computations the framework template might consider "metrics" — the file-01 Reminder-to-Confirmation rate and the `BR-008` Adherence percentage — are already owned by file 01 as success metrics, by `BR-008` in `_context/02_domain_model_and_business_rules.md` as a business-rule calculation, by `UC-009` in `_context/03_user_experience_and_use_cases.md` as a Patient-facing screen, and by the `AdherenceCalculator` in `_context/04_solution_architecture.md` as a pure-Domain function. They are **product features**, not observability indicators, and are out of scope for this section.

- **Tracing or correlation rules:** `Not applicable`. NemoPill is a single-process, single-device, no-network application — no distributed tracing surface exists. The framework template's "request IDs / job IDs / correlation IDs" vocabulary is server-side scaffolding that has no clientside analog here. Intra-app, every domain event, every `Dose`, and every `Confirmation` already carries the relevant aggregate IDs (`MedicationId`, `DoseId`, `DoseScheduleId`, `ConfirmationId`); a developer chasing a bug uses those IDs directly, not a synthetic correlation-ID layer.

- **Alerting expectations:** `Not applicable` for normal-operation alerts — there is no ops team, no on-call rotation, no PagerDuty-equivalent surface for a solo-dev local-only app, and the Patient is the wrong audience for an "alert" in the framework-template sense. Unexpected-failure reporting (Firebase Crashlytics or a comparable Android crash reporter) is acknowledged here as a **deferred** concern and is locked in `_context/12_environments_and_devops.md`: when it lands, the Developer is alerted via the crash-reporter dashboard for unhandled exceptions and `Result.Err.Unexpected`-flagged failures during pre-release testing and post-release operation. The Patient is never alerted by a runtime "something is wrong" channel — Patient-facing communication about misconfiguration (e.g., revoked `SCHEDULE_EXACT_ALARM` permission, disabled notifications) is a product-UX concern that lives in `_context/03_user_experience_and_use_cases.md` and `_context/11_visual_language_and_design_system.md`, not in observability.

### Notes on observability for a single-process clientside app

- The framework template's vocabulary (latency, throughput, queue depth, on-call alerting, distributed tracing) was written for server-side systems. Most of it maps to `Not applicable` here. What remains meaningful and is captured above: (a) a minimal logging policy that keeps logcat low-noise while still surfacing real surprises, and (b) a deferred crash-reporting hook in file 12 that closes the only legitimate "Developer needs to know about an unexpected failure" gap.
- **No analytics SDK in MVP.** The product handles sensitive personal data (medication identity, schedule, Adherence history) and file 01 explicitly says that data must not be transmitted or shared without explicit Patient consent. Adding any third-party analytics or telemetry SDK (Firebase Analytics, Mixpanel, Amplitude, etc.) is **out of scope for MVP**. This is intentionally restated in this section so that the Developer reading file 05 cannot inadvertently introduce such a dependency without first revising file 01.

## Security Guardrails

| Area | Rule | Enforcement Point |
| --- | --- | --- |
| Authentication and authorization | The Android OS lock screen is the **primary** authentication boundary — anyone with physical possession of an unlocked device is treated as the Patient. On top of that, the app exposes a **Patient-configurable app-level gate**: when enabled, `androidx.biometric.BiometricPrompt` (biometric or device-credential fallback) must succeed before the app surfaces any Medication / Dose / Confirmation data, and the gate re-engages after a Patient-configurable inactivity timeout. Default: off. Both the on/off and the timeout live in Settings. No per-screen or per-action re-authentication. No multi-user, no caregiver, no role-based authorization — file 01 puts those out of scope. | `MainActivity.onCreate` / `onResume` route through a `BiometricGateScreen` Composable in `:app::presentation` when the gate is enabled. Gate state (enabled flag and timeout) lives in app-private storage; the OS-handled biometric material itself is managed by `BiometricPrompt` and the Android Keystore by API contract. An architecture-conformance test asserts that no screen except `BiometricGateScreen` is reachable when the gate state is "locked." |
| Input validation | **Defense-in-depth.** Canonical validation lives in Domain — every value object (`Strength`, `Form`, `TimeOfDay`, `DateRange`, `Frequency`, `DoseQuantity`) enforces its preconditions in its constructor, and every Application use case enforces the relevant business rule (`BR-002` date range, `BR-003` zero/negative quantity, `BR-005` Confirmation enum bounds, `BR-011` retroactive 24-hour window, plus the `BR-001` / `BR-006` / `BR-007` aggregate invariants). On top of that, the Presentation layer duplicates format / range checks in ViewModels so the UI renders an inline error synchronously rather than round-tripping through `Result.Err`. Presentation never *substitutes* for Domain validation — duplication is intentional. Untrusted input arriving from OS callbacks (notification-action `PendingIntent` extras, `BroadcastReceiver` intent extras) is parsed into typed values **at the entry point** in `:presentation`, not at the use-case boundary. | Domain value-object constructors and Application use cases are the canonical enforcement. Parallel format-check helpers in `:presentation` ViewModels handle inline UX. A Konsist / ArchUnit test asserts that no Domain method swallows or relaxes a documented precondition. |
| Secret management | **No secrets in source code** — zero API keys, zero OAuth tokens, zero backend credentials, zero embedded passphrases (there is no backend by design). Release-signing keys are managed locally by the developer outside the repository; their storage convention is locked in `_context/12_environments_and_devops.md`. **No runtime secrets exist in MVP** — the biometric-gate enabled-flag and timeout from the Authentication row are configuration state, not secrets. **Forward-looking rule:** if a future product decision introduces a local-only runtime secret (e.g., an export-encryption key, an opaque token for a future server-side feature), it must be persisted via `androidx.security.crypto.EncryptedSharedPreferences` or directly in the Android Keystore — never plain `SharedPreferences`, never a plain Room column, never an in-memory `companion object` constant. | Build-time secret scanner (gitleaks, truffleHog, or equivalent — vendor TBD in file 12) runs on every commit and fails the build if a credential-shaped string is detected in source. The forward-looking rule for runtime secrets is asserted by a Konsist / ArchUnit test that flags any use of the unencrypted `SharedPreferences` API surface for keys matching a sensitivity pattern, *when* such keys exist. |
| Data protection | Three composing rules. **(a) Plain-text local storage** — the Room database is **not** encrypted (no SQLCipher in MVP). The Android app-sandbox is the protection boundary; under the device-locked threat model that underpins the Authentication row, encrypted-at-rest yields marginal incremental protection at the cost of a real dependency, a non-trivial cold-start, and a passphrase-management problem. **(b) No off-device transmission** — the app declares **no `android.permission.INTERNET`** in the manifest, and the codebase contains no HTTP / WebSocket / gRPC / DNS client code. This is the bullet that operationalizes file 01's commitment that medication data must not be transmitted or shared without explicit Patient consent. **(c) Patient-controlled hard deletion** — a Settings action ("Delete all my data") drops every Room table and resets app state. **Deletion is irreversible — there is no recovery, no backup, no undo.** Before the action runs, the Patient is shown a confirmation dialog that states this in plain language ("This will permanently delete every Medication, Dose, and Confirmation. NemoPill cannot recover this data afterward — there is no backup.") and requires an explicit, deliberate confirmation step (typed phrase or two-step tap pattern, locked in file 11), not a single tap on "OK." **No export feature in MVP** — exporting medication data to a file that could then be accidentally shared is itself a data-protection risk; file 01 keeps data on-device. | (a) Room database declared in `:app::di::AppModule` with no SQLCipher dependency on the classpath. (b) `AndroidManifest.xml` declares no `android.permission.INTERNET`; a Konsist / ArchUnit test asserts (i) the permission is absent and (ii) no class in any module imports `java.net.*`, `okhttp3.*`, `retrofit2.*`, or any analogous network API. (c) The Settings deletion flow lives in `:app::presentation::settings`; the confirmation copy and the explicit-confirmation interaction pattern are locked in `_context/11_visual_language_and_design_system.md` so the deletion UX cannot be silently softened later. |
| Dependency and supply chain | **Manual review.** Every new Gradle dependency (direct, *or* a meaningful transitive update) is reviewed by the solo developer at the moment it lands. No automated CVE scanning in MVP. New dependencies and version bumps require an explicit commit whose message includes a one-line justification (`why this lib, what it replaces, where it's used`). A `gradle/libs.versions.toml` change is treated as a dependency-review event by the Developer role and is a Security-role applicability trigger per the Quality Gates *Security guardrail review* row. | Solo-dev review at commit time. Any `gradle/libs.versions.toml` or feature-module `build.gradle.kts` change is flagged by the Security role's applicability check (Section 3 *Quality Gates*). Automated CVE scanning (`OWASP dependency-check` or equivalent) is deferred to file 12 and is the natural next step from this manual-review baseline once the dependency graph grows or once a regulated standard activates in the Compliance Applicability Matrix below. |
| OS permissions and IPC boundaries | **Minimum-permission manifest.** The app declares only the permissions named in file 04 — `SCHEDULE_EXACT_ALARM` or `USE_EXACT_ALARM` (final flavor in file 12), `POST_NOTIFICATIONS` on Android 13+, `RECEIVE_BOOT_COMPLETED`, and `WAKE_LOCK` if needed for the late-Reminder catch-up sweep. Explicitly **never** `INTERNET`, `READ_EXTERNAL_STORAGE`, `WRITE_EXTERNAL_STORAGE`, `READ_CONTACTS`, location, camera, or microphone. **Runtime-permission requests are surfaced at the right UX moment** — `POST_NOTIFICATIONS` and (where required by Android 13+) `SCHEDULE_EXACT_ALARM` are requested at the moment the Patient creates their first Medication, not at app launch. Every `BroadcastReceiver`, `Service`, and `Activity` in the manifest is declared with `android:exported="false"` unless there is a documented reason otherwise; `BootCompletedReceiver` is the one expected exception (`exported="true"` is required for the OS to deliver `BOOT_COMPLETED`), and that exception is annotated inline in the manifest. **All `PendingIntent` instances** (the inline-action "Taken" / "Skipped" notification actions and every other `PendingIntent` constructed by any module) are created with `PendingIntent.FLAG_IMMUTABLE` on Android 12+; this is mandatory, not advisory — a mutable `PendingIntent` would allow intent-redirection from any other app on the device. | `AndroidManifest.xml` declares the permissions and `exported` flags. A Konsist / ArchUnit test asserts (i) no permission outside the allow-list is declared, (ii) no component is `exported="true"` outside the documented `BootCompletedReceiver` exception, (iii) every `PendingIntent.getBroadcast` / `.getActivity` / `.getService` call site uses `FLAG_IMMUTABLE`. The runtime-permission flow lives in `:app::presentation::permissions` and is exercised by an integration test under Robolectric. |

### Notes on what's deferred out of Security Guardrails

- **Secret-scanner vendor** (gitleaks, truffleHog, or equivalent) — `_context/12_environments_and_devops.md`. The rule is locked here; the tooling is locked there.
- **`SCHEDULE_EXACT_ALARM` vs. `USE_EXACT_ALARM` flavor decision** — file 12, per the deferral already recorded in file 04.
- **Automated CVE scanning** (`OWASP dependency-check` or equivalent) — explicitly *deferred*, not committed. It is the natural next step from the manual-review baseline once the dependency graph grows or once a regulated standard activates.
- **Threat model** — `_context/13_threat_model_and_data_classification.md` will expand on the device-locked threat model that underpins the Authentication row's "OS lock screen is the primary boundary" claim and the Data protection row's plain-text-Room decision. The guardrails above are the *rules*; file 13 captures the *adversaries and assumptions* those rules are calibrated against.
- **No export feature** is recorded as a **deliberate omission**, not an oversight. If a future product decision introduces an export affordance, it requires (a) a revision of file 01's scope to permit Patient-driven sharing, (b) a Security-role review of the export format and the OS-level sharing surface, and (c) a new row or revision in this table covering export-time encryption and the confirmation UX. The same gate applies to any future import feature.

## Evidence And Audit Trail

- **Required artifacts to preserve:** Seven artifact classes, all stored in the project's single GitHub repository at `https://github.com/izirodriguez/NemoPill`. There is no external store of record, no separate evidence locker. Artifacts group into three buckets — design / decision trail, compliance / regulatory, and dependency review.
    
    **Design and decision trail.**
    - **The `_context/` framework files themselves.** The contextualized SDF files (`_context/01` through `_context/09` plus whichever optional files exist) *are* the design audit trail. Their git history is the change record; no separate artifact is derived from them.
    - **ADRs in `_context/09_decision_log.md`.** Every durable architecture, policy, integration, security, or delivery decision lands as an ADR per `CLAUDE.md`. The ADR text is the artifact.
    - **Role-turn entries in `_context/handoff_log.md`.** Every Architect / Project Manager / Developer / QA / Security role turn produces a structured entry, including the Section 3 *Quality Gates* manual-gate verdicts (`Coverage Sufficient`, `Guardrail Pass`, `_context/` consistency check). The log is the role-handoff audit trail; chat history is intentionally **not** preserved as a separate artifact because the handoff log captures the durable summary.
    - **Active-task-packet git history at `_context/08_active_task_packet.md`.** Every task starts at `Task status: Ready for proposal`, transitions to `Approved for apply` (human-only), and concludes when the next task replaces it. The git history of that single file is the per-task lifecycle record — each commit captures one state transition or one revision.
    
    **Compliance and regulatory artifacts.**
    - **Privacy Policy version history.** The bilingual (English + Spanish) Privacy Policy required by the Section 6 *CCPA / CPRA* row and the *Latin American national data-protection laws* row lives in a top-level `legal/` directory in the repo (`legal/PRIVACY_POLICY.en.md` and `legal/PRIVACY_POLICY.es.md`). Versioning is by git tag — each public version of the policy is tagged (e.g., `privacy-policy-v1.0`, `privacy-policy-v1.1`) so "what was the Privacy Policy on date X" is answerable by checkout. The Play Console upload is a copy of the tagged version, not a separate source of truth.
    - **Play Console submission snapshots.** Every meaningful Play submission (initial publish, version update, Data safety declaration change, country-availability change, category change) is mirrored back to the repo as a snapshot at `_build/play-store-state.md`, updated in the same commit as any change that requires the Play Console to be re-touched. The git log of that file is the Play-state-over-time audit trail.
    
    **Dependency review.**
    - **Dependency-review commit messages.** Section 5 § Dependency and supply chain requires that every new Gradle dependency (or transitive update worth reviewing) commits with a one-line justification in the commit message — `why this lib, what it replaces, where it's used`. `git log -- gradle/libs.versions.toml` is the dependency-review artifact; no separate dependency-review document is maintained.
    
    **What is intentionally not preserved.** Build logs (`./gradlew test` output), local IDE caches, coverage HTML reports, snapshot diff renderings, and ephemeral CI artifacts (when CI lands per file 12) are **not** preserved as evidence — they are regenerable from any commit, and persisting them adds noise without adding signal. The committed snapshot baselines under `:src/test/snapshots/` (or wherever Roborazzi / Paparazzi lands them per file 11 / file 12) *are* in-repo and are preserved by extension; the regenerable HTML reports of those tests are not.

- **Retention expectation:** Indefinite for in-repo artifacts; the repository itself must remain accessible for as long as NemoPill is distributed on the Play Store, plus a regulator-prudent tail.
    
    **In-repo retention.** Because every artifact above lives in `https://github.com/izirodriguez/NemoPill`, retention is the git history's retention — indefinite by default. Force-pushes that rewrite history on `main` are out of policy; `CLAUDE.md`'s rule that destructive git operations require explicit human instruction is the procedural guardrail behind this.
    
    **Repository accessibility.** The repository must remain accessible for as long as NemoPill is published on the Play Store, plus a **regulator-prudent tail of five years** after distribution ends. The five-year tail is calibrated to the longest plausible statute-of-limitations / inspection window across the in-scope jurisdictions (CCPA / CPRA enforcement actions, Latin American regulators' inspection windows, FDA general retention guidance for non-regulated wellness products). "Accessible" means the developer can clone the repo on demand; it does not require continuous public visibility. If the repo is ever made private or archived, the developer retains a local mirror with the same accessibility commitment.
    
    **External dependencies on third-party retention.** Two artifacts referenced in this section live outside the developer's direct control: the **Play Console** retains its own submission history under Google's policy (which the developer cannot modify), and **GitHub** retains the repository under GitHub's terms (which the developer also cannot modify). The in-repo `_build/play-store-state.md` mirror exists specifically so that the Play Console's audit trail is *also* captured in the developer-controlled repository — if Google's retention changes or the Play Console account lapses, the in-repo mirror remains.
