# Solution Architecture

## How To Fill This File

- Describe the intended shape of the solution, not every incidental implementation detail.
- Keep the dependency rules crisp enough that an engineer can decide where code belongs.
- Default to functional composition and explicit interfaces over hidden framework coupling.

## Architecture Summary

- Architecture style: Clean Architecture.
- Deployment shape: Single Android application — installed on the Patient's device; no backend, no cloud sync, no separate services. Background scheduling runs in-process via Android's WorkManager / AlarmManager.
- Primary runtime environments: Android mobile (foreground UI process) and Android OS-managed background runtime (`BroadcastReceiver`, `WorkManager`, `AlarmManager` callbacks).
- Key architectural constraint: Reminder reliability under Android background-execution restrictions (Doze, App Standby, battery optimization, reboot). The Scheduler must fire each Reminder exactly once at its scheduled time across these conditions (BR-004), or the product fails its primary purpose.

### Notes on cross-module communication and the `scheduling` ↔ `notifications` boundary

- **Cross-module communication is event-driven.** A feature module never imports another feature's Domain or Application packages. Modules emit domain events on `DomainEventPublisher` (in `:core`) and other modules subscribe.
- **`scheduling` ↔ `notifications` boundary:** `scheduling` decides *that* a Reminder must fire; `notifications` decides *what the Patient sees*. BR-010's distinct "Late: …" text and the silent-missed / batched-summary copy are presentation concerns owned by `notifications`. The clock arithmetic that decides on-time vs. late vs. missed lives in `scheduling`.
- **Event transport for MVP is in-process** (per file 02 handoff): a Kotlin Coroutines `MutableSharedFlow<DomainEvent>` in `:core`. No message bus, no Room-as-outbox.

### Notes on what's *not* a port

- **Use cases (interactors) are not ports themselves** — they are the Application layer's public API consumed directly by Presentation (`ViewModel`s, `BroadcastReceiver`s, `WorkManager` Workers). No interface between Presentation and Application is needed for testability since use cases are already small, single-purpose classes.
- **`BroadcastReceiver` and `WorkManager.Worker` classes are entry points (Presentation layer per the layer table), not adapters.** They translate OS callbacks into use-case invocations.
- **Database migrations** are not a port — they live inside the Room adapter modules and are coordinated via Room's `Migration` API. May be revisited if a `:shared:test-fixtures` module is added in file 12.

## Functional Programming Rules

- **Prefer pure functions for business logic.** A Domain function counts as pure when (a) every dependency it needs — including the current time and the device timezone — is passed in as a parameter or constructor argument (no `System.currentTimeMillis()`, no `ZoneId.systemDefault()`, no service locator, no global state); (b) it performs no I/O (no Room, no `AlarmManager`, no notification, no logging); (c) it has zero Android imports. **Verification rule:** a Domain unit test must compile and run on a plain JVM with only `kotlin-stdlib` + `junit` + `truth` on its classpath — if a Domain class needs `android.*` to compile, it's miscategorized.
- **Immutability rule.** Domain entities and value objects are Kotlin `data class` with `val` properties only. State changes are expressed by returning a **new instance** — e.g., `medication.archive(now: Instant): Medication` returns a new `Medication` with `isActive = false`; the original is never mutated. Aggregate operations return either a new aggregate snapshot or a sealed `Result<NewState, BusinessRuleViolation>`. Collections in Domain types are immutable (`List<T>`, `Set<T>`, `Map<K, V>` — never the `Mutable*` variants). The only mutable state in the system lives **inside Infrastructure adapters** (Room's mutable cursor, `MutableSharedFlow`, etc.) and is invisible above the port boundary.
- **Side-effect isolation rule.** All I/O — persistence, clock reads via `ClockPort`, timezone reads via `TimezonePort`, scheduling via `SchedulerPort`, notifications via `NotificationPort`, domain-event emission via `DomainEventPublisher` — is invoked **only from the Application layer**. Application use cases orchestrate effects and Domain calls; the body of every Domain function is pure. Practical consequence: **Domain functions are non-suspending by default**. `suspend` modifiers appear in Application use case signatures and Infrastructure adapter implementations, never on Domain entity methods. The `pending → missed` evaluation rule is a Domain function that takes `now: Instant` as a parameter — it is not allowed to ask `Clock` itself.
- **Error handling rule.** **Two-tier policy.**
  1. **Expected failures** (BR-002 invalid date range, BR-003 zero/negative quantity, BR-011 retroactive window expired, BR-001 atomic-replace conflict, BR-004 Reminder-fire skip due to status change, BR-007 archive-with-future-pending atomicity failure, etc.) are represented by an explicit `Result<T, E>` sealed type defined in `:core`. Use cases return `Result`; Presentation pattern-matches on the sealed sub-types and surfaces a non-technical message per file 03's UX contract. **No expected business failure is signaled by an exception.**
  2. **Unexpected failures** (Room database corruption, framework misuse, `null` where the type system says non-null, OS API contract violations) propagate as exceptions. They are caught at the Application/Infrastructure boundary, logged, and translated into a generic `Result.Err.Unexpected` for the UI. They are *never* silently swallowed.

  Domain value-object constructors are allowed to throw `IllegalArgumentException` for true precondition violations the caller should have prevented (e.g., `Strength(amount = -1)`); calling code at the Application boundary catches and converts to `Result.Err`. This keeps Domain construction cheap while ensuring the boundary always returns a typed result.
- **Shared utility rule.** Cross-cutting helpers — `Result<T, E>`, ID newtypes (`MedicationId`, `DoseId`, `DoseScheduleId`, `ConfirmationId`), `ClockPort`, `TimezonePort`, `DomainEventPublisher`, the `DomainEvent` sealed hierarchy itself — live in the `:core` module's shared-kernel packages. **Two admission rules:** a helper enters the shared kernel only if (a) at least two of the three feature modules genuinely need it, and (b) it has zero Android dependencies (`:core` is built as a `kotlin("jvm")` Gradle module so the build itself enforces "no Android imports" — feature Domain layers can therefore reference any shared-kernel type without violating the layer table's "Domain must not depend on Android SDK" rule). **No `util` package** inside feature modules — every shared abstraction has to earn its keep with a name that describes its single purpose. Helpers specific to one feature stay scoped `internal` to that feature module.

### Gradle dependency graph

```text
:app  ──► :core
      ──► :medication-management
      ──► :scheduling
      ──► :notifications
      ──► :adherence-tracking

:medication-management  ──► :core
:scheduling             ──► :core
:notifications          ──► :core
:adherence-tracking     ──► :core

:core                   ──► (nothing — pure Kotlin + java.time only)
```

Feature modules **do not** depend on each other and **do not** depend on `:app`. Cross-Bounded-Context communication goes only through `DomainEventPublisher` (in `:core`). `:app` is the only module allowed to depend on the four feature modules — and only for DI wiring, navigation, and the cross-cutting screens.

## Architecture Risks And Trade-Offs

- **Trade-off accepted: layer enforcement at compile-time only at the *module* boundary, not within a module.** The Gradle module graph guarantees that a feature module can't accidentally import another feature's Domain — the only legal cross-Bounded-Context path is a domain event on `DomainEventPublisher`. **Inside** a feature module, however, the four-way split (`domain/` / `application/` / `infrastructure/` / `presentation/`) is enforced only by package convention, Kotlin `internal` visibility, and code review — there is no compile-time check that, say, `domain/` doesn't import `infrastructure/`. For a solo dev this is a deliberate cost-control choice; if accidental layer creep appears, the cheapest counter-measure is a Konsist or ArchUnit test in `:core` rather than splitting each feature into multiple sub-modules. **Revisit if:** a feature module grows past ~40 classes or layer drift shows up in code review.
- **Trade-off accepted: in-process domain events (`MutableSharedFlow`) with no durability.** Cross-module reactions (e.g., `notifications` reacting to `ReminderFired`, `adherence-tracking` reacting to `DoseMissed`) only fire while the app process is alive. The two cases where this *could* matter are masked by other architecture choices: (a) Reminders firing while the app is dead is handled by `AlarmManager` / `BootCompletedReceiver` re-registration, not by event replay; (b) BR-008 Adherence is computed on demand, so missed event deliveries don't corrupt history. **Revisit if:** a future feature requires reliable event delivery across process death (e.g., "notify the Patient when their 7-day Adherence first drops below 50%"). The migration path is Room-as-outbox + a worker that drains pending events.
- **Trade-off accepted: the `Dose` aggregate's transaction spans *two* Gradle modules.** `Dose.status` lives in `:scheduling` (it's part of the Dose lifecycle) but `Confirmation` lives in `:adherence-tracking`. The Dose-aggregate atomicity from file 02 — "Confirmation write and Dose status update happen in the same transaction" — is preserved by routing both writes through a single Room `withTransaction { }` block coordinated by `adherence-tracking`'s `ConfirmDoseUseCase`, which depends on `DoseRepository` (a port from `:scheduling`) in addition to its own `ConfirmationRepository`. **The strain shows here:** if we ever moved `:scheduling`'s and `:adherence-tracking`'s Room databases into separate `RoomDatabase` instances (e.g., for partitioning or per-feature versioning), the same-transaction guarantee breaks. **Mitigation:** keep a single `NemoPillDatabase` instance for the whole app, declared in `:app::di::AppModule`, with all four feature DAOs registered on it. **Revisit only if:** there's a strong reason to partition the Room database, which there isn't for MVP.
- **Refactor candidate: in-place Confirmation mutation (Option A) loses the audit trail.** Per file 02's resolution, a Confirmation correction overwrites `status`, `confirmedAt`, and `source` in place. This was the right call for MVP simplicity, but it means the system can't answer "did this Patient actually back-edit yesterday's Dose, and if so, what did they change?" If a future iteration wants Confirmation history, suspicious-back-edit detection, or any kind of forensic view, the migration is to switch to **event-sourced Confirmations** (an append-only `ConfirmationEvent` table; current state derived) or to add a `ConfirmationHistory` audit table written transactionally alongside the in-place update. **Revisit if:** the Patient's clinician asks for the data, or if Adherence trust becomes a question.
