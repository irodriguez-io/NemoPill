# Delivery Plan And Milestones

## How To Fill This File

- Break work into milestone slices that can be reviewed, demonstrated, and approved in order.
- Keep milestones outcome-oriented rather than team-activity-oriented.
- Use stable IDs so `08_active_task_packet.md` can point to the current milestone and task unambiguously.

## Delivery Summary

- Current delivery phase: **`MVP — design (pre-code)`**. NemoPill is currently inside the design-phase half of file 01's success metric — all `_context/` files are being contextualized before any code is written. The phase label flips to `MVP — implementation` once `_context/` is complete and the validator passes (per file 01's design-phase exit criterion).
- Release cadence: **`Per milestone`**. Each milestone closes with a self-built debug APK installed on the developer's own device for demonstration. No Google Play submission in MVP. Play Store distribution work (internal testing track, *Data safety* declaration, bilingual Privacy Policy upload) is scoped to a later phase and is owned by `_context/12_environments_and_devops.md` per the file 05 / file 06 handoffs.
- Demo or review cadence: **Per task packet + end-of-milestone retrospective review.** Every task packet (`_context/08`) goes through the two-pass approval flow defined in file 05 — Developer proposes BDD scenarios → QA renders `Coverage Sufficient` / `Coverage Insufficient` verdict → human flips `Task status` to `Approved for apply`. In addition, the close of each milestone triggers a retrospective review: confirm the milestone's Done When criteria are met, run the file-05 quality gates (incl. coverage thresholds and architecture-conformance tests), update affected `_context/` documents, and append the ADR(s) the milestone produced.
- Known calendar constraints: **Human gatekeeper review-and-approval bandwidth ≈ 3–4 hours per week (evenings and weekends).** Implementation work is agent-driven (Project Manager, Developer, QA, Security, Compliance, Legal roles are filled by Claude Code); the constraint is the Human gatekeeper's review and apply-approval cadence, not single-handed coding throughput. No fixed external deadline, no audit date, no app-store submission deadline in MVP. Milestone durations are therefore wall-clock loose by design — the milestones are scoped by *outcome*, not by *date*, and the plan optimizes for a reviewable slice rather than a calendar-week increment. If real-world review bandwidth drops below this baseline for an extended period, the right response is to re-scope the *active* milestone in `_context/08`, not to recalibrate the milestone register globally.

## Milestone-Level Definition Of Done

- **The milestone goal is demonstrably met:** every Done When item in the milestone's row of the Milestone Register above is satisfied with auditable evidence (test pass, screenshot, log line, file present in repo, or hardware-test note in `_context/handoff_log.md`). The Project Manager role assembles the close-out evidence package; the Human gatekeeper confirms goal satisfaction before flipping the milestone Status from `Active` to `Complete`. For milestones with user-visible behavior, the close-out evidence includes installation of the milestone's debug APK on the developer's hardware and a walk-through of the affected Key Journey(s) from `_context/03`.
- **The relevant tests and quality gates pass:** all file-05 quality gates are green for the milestone's scope — type check + lint + fast test suite (pre-commit), slow test suite + coverage + debug build + framework validator (pre-merge), QA two-pass TDD review (pre-Apply-Mode), Security guardrail review and `_context/` consistency check (task close). Coverage thresholds from file 05 hold across every module touched by the milestone (`:domain` and `:core` ≥ 90 %, `:application` ≥ 80 %, no `%` target on `:infrastructure` and `:presentation`). Architecture-conformance tests (Konsist or ArchUnit) remain green, including the no-`android.permission.INTERNET` and no-network-imports assertions, and the `PendingIntent.FLAG_IMMUTABLE` assertion. The **two-pass TDD discipline** from file 05 holds for every task in the milestone: the Developer role proposes BDD scenarios referencing file-03 BDD IDs and file-02 BR rules in `_context/08_active_task_packet.md`; the QA role renders a `Coverage Sufficient` or `Coverage Insufficient` verdict; only after `Coverage Sufficient` may the Human gatekeeper flip `Task status` to `Approved for apply`. Neither QA nor Developer can self-approve, and the Human cannot flip to `Approved for apply` while QA's verdict is `Coverage Insufficient`. `bash _framework/validate_framework.sh` exits zero at milestone close.
- **User-facing or operational behavior is reviewed:** the Human gatekeeper walks each Key Journey from `_context/03` affected by the milestone using the installed debug APK. For **M-002 and M-004**, the walk-through includes a manual Doze test on real Android hardware — a Reminder scheduled before the device enters Doze must still fire at the configured time (file 01 assumption (1) verified in production conditions, not just under Robolectric). For **M-006**, the walk-through includes a TalkBack accessibility pass against the file-03 environment constraints (TalkBack-compatible, low-light glanceable, one-handed friendly, battery / Doze tolerant) and a responsive-layout check at the phone / tablet / foldable breakpoints from file 03. Where file-05 Snapshot tests are part of the milestone scope, the Human reviews the snapshot diffs (new or changed baselines) before accepting them as the new visual contract.
- **Documentation updates are complete:** ADRs introduced during the milestone are appended to `_context/09_decision_log.md` before close (if no new ADRs surfaced, the milestone-close handoff entry explicitly records "no new ADRs this milestone"). Every `_context/` file whose documented truth changed during implementation is updated and re-validated by `bash _framework/validate_framework.sh` (zero unresolved framework placeholders and zero inline review annotations; validator exits zero). A **milestone-close entry** is appended to `_context/handoff_log.md` covering: (i) what was delivered; (ii) which Business Rules (`BR-xxx`) and OS-crossing flows (`F-xxx`) are now implemented and tested; (iii) the cumulative ADRs added during the milestone (referencing file 09 entry IDs); (iv) any open questions or inline review annotations carried forward; (v) the next milestone's expected entry conditions. For milestones whose scope touches the file-05 § Evidence chain (notably M-006), the relevant repo artifacts (`legal/PRIVACY_POLICY.en.md`, `legal/PRIVACY_POLICY.es.md`, `_build/play-store-state.md`) are committed at the state agreed in the milestone's Done When cell.

## Task Slicing Rules

- One task packet must describe exactly one task that can be reviewed in a single pass.
- A task must belong to one milestone only.
- A task should produce one coherent user-visible, domain-visible, or operational outcome.
- If a task needs more than one approval cycle, split it into smaller tasks before giving it to the agent.

## Review And Approval Rules

- **Proposal review owner:** **QA role (Claude Code) renders the Coverage verdict** on the Developer role's proposed BDD scenarios in `_context/08_active_task_packet.md` — `Coverage Sufficient` or `Coverage Insufficient`, referencing file-03 BDD IDs and file-02 Business Rule IDs (`BR-001` … `BR-012`). **Security role (Claude Code) renders a guardrail review** in parallel when the task surface touches `_context/05` in-scope data, controls, or guardrails (manifest permissions, IPC boundaries, `PendingIntent.FLAG_IMMUTABLE` patterns, Room schema changes, any apparent network-stack import, the Play *Data safety* declaration), or any file-06 trust boundary. Both reviews must render `Coverage Sufficient` and `Guardrail Pass` (where Security is activated) before the Human gatekeeper may flip `Task status: Approved for apply`. **Compliance and Legal are folded into the Security review** per file 05's Compliance Activation Rule — although several rows in the Compliance Applicability Matrix are `Applicable` (CCPA / CPRA; Latin American national data-protection laws), no separate Compliance or Legal seat is activated; Security cites the relevant control or evidence requirement on tasks that touch those rows. QA cannot self-approve. Developer cannot bypass QA. Security can block Apply Mode independently of QA per the CLAUDE.md role contract.
- **Apply approval owner:** **Human gatekeeper (Isidro)** is the only role authorized to set `Task status: Approved for apply` in `_context/08_active_task_packet.md`. This is the load-bearing human checkpoint in the framework — the agent roles propose, review, and verify; the Human alone authorizes implementation. Per file 05's two-pass TDD discipline, this flip is permitted only after QA has rendered `Coverage Sufficient` and Security has rendered `Guardrail Pass` (where activated). The Human may not flip to `Approved for apply` while either review is in a blocking state. The Project Manager role explicitly may not advance `Task status` to `Approved for apply` per CLAUDE.md.
- **Completion sign-off owner:** **Human gatekeeper (Isidro)** for both task close and milestone close. At **task close**, the Project Manager role assembles the evidence package (tests pass, coverage thresholds still held across affected modules, architecture-conformance still green, file-06 F-flow citations satisfied, conditional Security guardrail review at task close per file 05); the Human reviews the package and flips `Task status: Complete`. At **milestone close**, the Project Manager role assembles the milestone-level evidence package per the Milestone-Level Definition Of Done above (every Done When item from the Milestone Register satisfied, all relevant Business Rules and OS-crossing flows tested, snapshot diffs accepted, debug APK installed and demoed, Doze / accessibility / responsive-layout checks completed where applicable, ADRs backfilled in `_context/09_decision_log.md`, milestone-close handoff entry appended to `_context/handoff_log.md`); the Human reviews the package and flips the milestone Status from `Active` to `Complete` in the Milestone Register. Sign-off is the deliberate human moment that closes one reviewable slice and opens the next — it is distinct from goal-satisfaction verification, which is automated and role-assembled.
