## Task Objective

- **Problem to solve now:** T-011 closed the observe leg — the on-screen counter recomposes 0 → 1 when a `Confirmation` is persisted — so the M-002 vertical now runs end-to-end **while the process lives**. But `AlarmManager` exact alarms are **cleared by the OS on reboot**, and nothing in the app persists the demo Reminder's scheduled time, so a device restart during the 10-minute wait silently drops the Reminder. M-002 Done When item (6) requires that "`BootCompletedReceiver` is registered and a manual reboot during the 10-minute wait re-arms the alarm (F-011 happy path)." The `BootCompleteReceiver` scaffold exists in `:app` (`io.nemopill.app.boot.BootCompleteReceiver`) but its `onReceive` is a no-op (`= Unit`), and it is already manifest-declared (`exported="true"`, `BOOT_COMPLETED` filter) with `RECEIVE_BOOT_COMPLETED` already granted. T-012 makes the receiver actually re-arm the demo alarm.
- **Why now:** item (6) is the **last remaining code task** in M-002. After T-011, the only open Done When items are item (2)-hardware (the manual real-device Doze test — a milestone-close gate, not a task) and item (6). Closing item (6) makes M-002 eligible for milestone-close sign-off (which still requires the manual hardware reboot/Doze test and the gatekeeper's `_context/07` `Active → Complete` flip — neither is a T-012 edit). Boot-survival is also the natural next slice: it builds directly on the merged T-008 scheduling adapter and proves the final leg of file-01 assumption (1) (Reminders survive the OS lifecycle, including reboot).
- **Expected user-visible / system-visible outcome:** After T-012 lands and merges, the manual demo is: tap "Schedule the Demo Reminder" (arms `AlarmManager.setAlarmClock` for `now + 10 min` via `AlarmManagerSchedulerAdapter`, **and now also persists the target `Instant`**); reboot the device before the 10 minutes elapse; on boot the OS delivers `BOOT_COMPLETED` to `BootCompleteReceiver`, which invokes `ReArmDemoReminderUseCase`, which re-reads the persisted target and — because it is still in the future — re-registers the same alarm through `SchedulerPort.scheduleReminder(DEMO_DOSE_ID, target)`; the Reminder then **fires at its original scheduled time**, delivering the notification (the T-009 path) exactly as if the reboot had not happened. `./gradlew :core:test :scheduling:test :notifications:test :adherence-tracking:test :app:test koverVerify` exits zero with the new tests present; all eight CI stages stay green; `arch-conformance` (Konsist) stays green (no new permission, no cross-feature import, `PendingIntent` immutability preserved). ### Architect Decision Required — the re-arm seam and the persistence mechanism

- **The decision (recorded; the Developer ratifies as an ADR at execution time):** Done When item (6) says a reboot "**re-arms the alarm**." `AlarmManager` alarms are cleared on reboot and nothing persists the demo target today, so there are two coupled choices:
  - **(1) Re-arm target.**
    - **(α) Re-arm to a fresh `now + 10 min`** on boot (the receiver just re-invokes `ScheduleDemoReminderUseCase`). Simplest — needs no persistence. **But** it **restarts the countdown** rather than re-arming the *original* alarm: a reviewer who reboots at minute 5 and expects the Reminder at the original minute 10 would instead wait a full 10 more minutes, and each reboot postpones the Reminder again. It does not honor "re-arms the alarm," and the manual hardware test would visibly show the wrong fire time.
    - **(C) ✅ Recommended — re-arm at the original persisted time.** Persist the demo Reminder's target `Instant` at schedule time; on boot re-read it and re-arm at that exact time if it is still in the future. This is what item (6) and the F-011 "boot-survival re-registration" literally require, it makes the manual hardware reboot test pass faithfully (the Reminder fires at its original time), and it **precedents the real M-004 boot path** (`_context/06 § F-011 / § F-012` re-arm persisted Doses at their stored `scheduledAt`).
  - **(2) Persistence mechanism for the target.**
    - **(α) SharedPreferences ✅ Recommended** — an app-private `SharedPreferences` file holding `doseId.value → epochMilli`. No new dependency, no Room schema change, trivially testable under Robolectric, sufficient for the single demo entry. Keeps T-012 a thin walking-skeleton slice.
    - **(β) Jetpack DataStore (Preferences)** — more idiomatic and `Flow`-friendly, but adds a **new dependency** and a new async seam for storing one `Long`; heavier than this slice warrants. Surface as the documented alternative if the Developer judges DataStore worth standardizing now (it will be needed eventually).
    - **(γ) A Room key-value table** — **rejected for T-012**: it forces a `NemoPillDatabase` schema change (`version` bump, new `2.json`, a migration) and a heavier Security/schema review, contradicting the "no new Room schema in the M-002 slices" ethos and AC-006. The real persisted-`Dose` re-arm is M-004's Room work, not a key-value hack here.
- **Consequence to record:** adopting **(C) + SharedPreferences** stands up the project's **first persisted scheduling state** and the **first at-rest `:scheduling` store**, behind a `PendingReminderStore` port so the M-004 multi-Dose boot path can swap the backing implementation (e.g. to the `Dose`/Room read path) without touching `BootCompleteReceiver` or `ReArmDemoReminderUseCase`. **Surface, do not silently resolve:** if the Developer finds (C) infeasible, fall back to (α) re-arm-fresh and record why; but prefer (C).
- **Layer-boundary check (no conflict):** `:app::boot` depending on `:scheduling::application` is **legal** — `:app` is the one module allowed to depend on every feature, and `:app` already declares `implementation(project(":scheduling"))`. The store I/O lives only in `:scheduling::infrastructure` behind the Application port (`NoUpwardLayerDependencyRule` green); `:scheduling` imports no other feature module (`NoCrossFeatureDomainImportRule` green). This is **not** a cross-feature seam.

- **Risks:**
  - **Boot-time receiver lifetime vs. the store read.** `onReceive` must finish quickly. *Mitigation:* mirror `ReminderAlarmReceiver` — `goAsync()` + a bounded `CoroutineScope` under `withTimeout(5_000)`, `pending.finish()` in `finally`. The SharedPreferences read is fast and runs on the background scope, not the main thread.
  - **Exact-alarm permission state at boot.** Re-arming via `setAlarmClock` assumes `USE_EXACT_ALARM` / `SCHEDULE_EXACT_ALARM` still holds. *Mitigation:* in M-002 the demo precondition is that the permission holds (it was granted to arm the original alarm); the revoked-at-boot UX affordance is explicitly M-004. The re-arm catches adapter exceptions and logs a static message rather than crashing the boot receiver.
  - **Past-due target at boot (device off > 10 min).** *Mitigation:* `ReArmDemoReminderUseCase` **no-ops** a target `<= now` (the F-008 missed-sweep is M-004). `AC-002` case (c) pins this.
  - **`ScheduleDemoReminderUseCase` ordering.** The store write must not mask a successful schedule nor double-fire. *Mitigation:* persist **after** `scheduleReminder` returns and **before** `Result.Ok`; a store-write exception maps to `Result.Err.Unexpected` (the alarm is already armed — acceptable for the demo; `AC-004` documents the ordering).
  - **Scope-creep toward M-004.** A reviewer might expect the full multi-Dose F-011 / missed-sweep. *Mitigation:* the packet is explicit — single-Dose happy-path stub only; recorded in the demo-vs-full gap ADR and the close handoff.
- **Blockers:** none. The Architect decision above is recorded with a recommendation; the Developer ratifies it as an ADR at execution time.
