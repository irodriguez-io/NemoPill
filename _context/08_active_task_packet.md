## Task Objective

- **Problem to solve now:** T-010 delivered the **persistence leg** of the M-002 walking skeleton — tapping "Take" / "Skip" writes a `Confirmation` row to `NemoPillDatabase` and dismisses the notification — but **nothing on screen reflects that write**. The demo screen still shows only the scheduling status line (`DemoUiState`); the `DemoViewModel`'s own doc comment records that "the Adherence counter and observing a `Confirmation` via `StateFlow` are M-002 item (5), out of scope for T-008." T-011 takes the **last functional leg** of the walking skeleton: turning the persisted `Confirmation` into a **visible, recomposing on-screen count**. This is M-002 Done When item (5) ("Compose screen observes the Confirmation via `StateFlow` and increments the hardcoded Adherence counter from 0 to 1"). It stands up the project's **first read/observe path** — a `Flow`-returning Room query, a query use case, and a ViewModel `StateFlow` collection — completing the milestone's stated end-to-end composition: "Compose → ViewModel → use case → Domain → port → adapter → `AlarmManager` → Doze-survival → `BroadcastReceiver` → `NotificationManager` → inline action → use case → Room → **`StateFlow` → Compose re-render**".
- **Why now:** the persistence leg (T-010) is merged and is the prerequisite — there is now a `Confirmation` row to observe. The natural next cut is making that write **glanceable** on screen. Slicing the observe leg on its own — a Room `Flow` count → `StateFlow<Int>` → recomposed counter, and nothing else — keeps a coherent, reviewable spine for the gatekeeper and isolates the project's first read path from the heavier BR-008 Adherence-calculation work (M-005). Per `_context/07 § Task Slicing Rules`, items (4) and (5) were deliberately separated so each fits one approval cycle; T-010 closed (4), T-011 closes (5). After T-011, the only M-002 Done When items left open are item (2)-hardware (the manual real-device Doze test — a milestone-close gate) and item (6) (`BootCompletedReceiver` re-arm — T-012).
- **Expected user-visible / system-visible outcome:** After T-011 lands and merges, launching the debug APK shows the demo screen with a counter reading **0**. Scheduling the demo Reminder, waiting for it to fire, and tapping **"Take"** writes the `Confirmation` (T-010) **and the on-screen counter recomposes to 1** (a "Skip" tap leaves the Taken counter at 0, since it persists a `SKIPPED` row; tapping "Take" after a "Skip" on the same demo Dose flips it back to 1 via the T-010 1:1-on-`doseId` REPLACE idempotency). `./gradlew :core:test :scheduling:test :notifications:test :adherence-tracking:test :app:test koverVerify` exits zero, including a new ViewModel `StateFlow` test, a use-case observe test, a Room `Flow`-count integration test, and a Compose/Robolectric test asserting the counter text recomposes to "1". All eight CI stages stay green; `arch-conformance` (Konsist) stays green (no new sensitive `data class`; ADR-049 rules (i) and (ii) untouched-and-green). A T-011-close handoff entry records the ratified observe seam, the demo-vs-full gap (the counter is a raw "Taken" tally, **not** BR-008 Adherence), and the next-role routing to T-012 (item 6).

### Architect Decision Required — what the counter observes (the `Confirmation` read seam)

- **The decision (recorded; the Developer ratifies as an ADR at execution time):** M-002 Done When item (5) says "Compose screen **observes the Confirmation** via `StateFlow`", while ADR-100 (T-010) named **T-011 the "first `DoseConfirmed` consumer."** These two framings pull toward two different surfaces:
  - **(α) Observe the `DoseConfirmed` event** off the `:core` `DomainEventPublisher` (a `:app` listener filters `DoseConfirmed`, increments an in-memory counter exposed as `StateFlow`). Matches ADR-100's literal wording. **Cost:** the count is **in-memory and non-durable** — it is lost on process death, and a cold-start replay (the `InProcessEventBus` `REPLAY = 8` buffer) forces manual dedupe on `(confirmationId, confirmedAt)` to avoid double-counting. It also does **not** exercise the "**Room → StateFlow → Compose re-render**" leg, which is the precise composition the M-002 milestone goal text exists to prove.
  - **(C) ✅ Recommended — observe the persisted `Confirmation` via a Room `Flow`** (a `ConfirmationDao` `@Query` returning `Flow<Int>`, surfaced through `ObserveConfirmedDoseCountUseCase`, collected into a `StateFlow<Int>` in the ViewModel). This is what M-002 item (5)'s "observes the Confirmation" and the milestone goal's "Room → StateFlow → Compose re-render" literally require; it is **durable across process death** (the realistic demo flow is *tap "Take" from the lock screen → open the app*, by which point the demo process may have been killed — a Room `Flow` re-reads the row on re-subscription, an in-memory event would be gone); and a SQL `COUNT` is **inherently idempotent**, so the replay/dedupe problem of (α) does not exist. It also directly **precedents the real M-005 / M-006 read path** (`_context/06 § F-013` `observeTodayDoses` is the same `Flow` → `StateFlow` → Compose shape).
- **Consequence to record:** adopting (C) **refines ADR-100's expectation** that T-011 is the "first `DoseConfirmed` consumer." `DoseConfirmed` is still **emitted** by T-010's use case but is **not consumed** in T-011 — its first real cross-module consumer moves to M-005 (Adherence-cache invalidation / recompute trigger). The Developer records this refinement in the T-011 ADR (it does not require editing ADR-100 — a new ADR that supersedes the relevant clause, per the `_context/09` "add a superseding ADR, don't rewrite history" rule). **Surface, do not silently resolve:** if the Developer finds (C) infeasible (it should not be — it is strictly simpler than T-010's seam), fall back to (α) and record why, but prefer (C).
- **Layer-boundary check (no conflict):** `:app::presentation` depending on `:adherence-tracking::application` is **legal** — `_context/04 § Module map` puts the cross-cutting Adherence surface in `:app`, `:app` already declares `implementation(project(":adherence-tracking"))`, and `_context/04 § layer table` line 57 has Presentation (`ViewModel`s) call Application use cases directly. This is **not** the contested cross-*feature* seam of T-010 (which was `:notifications` → `:adherence-tracking`); `:app` is the one module allowed to depend on every feature. `NoCrossFeatureDomainImportRule` is unaffected.

- **Risks:**
  - **Room `Flow` collection lifecycle vs. the demo wait window.** The counter must reflect a `Confirmation` written while the screen may be off-screen (the Patient taps from the lock screen). *Mitigation:* `stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), 0)` re-subscribes (and re-reads the durable Room row) when the screen returns to the foreground; because the truth is the Room row (decision C), the count is correct after any backgrounding or process restart — this is the central reason (C) beats (α). The `AC-005` Compose test and the `AC-004` ViewModel test pin the 0 → 1 behavior under deterministic emission.
  - **`status` enum round-trip in the `@Query` predicate.** The `WHERE status = :taken` parameter must match how `ConfirmationConverters` stores the enum (its `name` string). *Mitigation:* pass `ConfirmationStatus.TAKEN.name` (or a converter-typed parameter) and assert the round-trip in the `AC-003` integration test (write `TAKEN`, observe count `1`; write `SKIPPED`, observe `0`).
  - **`kotlinx-coroutines-test` on the `:app` test classpath.** The new `:app` ViewModel test needs `runTest`. *Mitigation:* add `testImplementation(libs.kotlinx.coroutines.test)` to `:app/build.gradle.kts` (the alias is already used by `:core` / `:adherence-tracking`, so no new version-catalog entry).
  - **Compose `WhileSubscribed` initial-value flake in tests.** A `stateIn` `StateFlow` starts at the initial value until the first upstream emission. *Mitigation:* the ViewModel test drives emissions deterministically with `runTest` + a controllable fake `Flow`; the Compose test renders the **stateless** `DemoScreen` with a fixed `confirmedCount` (no `stateIn` timing in the UI test).
  - **Scope-creep toward BR-008 Adherence.** A reviewer might expect a real Adherence percentage. *Mitigation:* the packet is explicit — the counter is a raw Taken tally, not BR-008; `ComputeAdherenceUseCase` and the percentage are M-005. Recorded in the demo-vs-full gap ADR and the close handoff.
- **Blockers:** none. The Architect decision above is recorded with a recommendation; the Developer ratifies it as an ADR at execution time.
