- Recovery from partial implementation: the most likely partial-implementation failure mode is conversation-context exhaustion before the ADR sequence is complete. Recovery has two paths: **(a) resume in a new conversation** — the new `/start-working` invocation will see `_context/09_decision_log.md` already partially present (containing the structural header sections plus the partial ADR sequence), run the framework validator to confirm the file's existence, read the mid-session handoff entry in `_context/handoff_log.md` to inherit context, and continue the ADR sequence from the next unwritten ADR per CLAUDE.md's per-file resume protocol. The packet's `Approved for apply` flag remains set, so the resume session can proceed directly to Apply Mode without re-litigating the inline-authorization caveat (the resume is a continuation of the same T-003 task, not a new task); **(b) full restart** — `git restore _context/09_decision_log.md`, abandon the partial sequence, restart fresh in a new conversation, possibly with revised granularity or detail-level choices if the partial draft revealed that the original choices were ill-calibrated. Path (a) is preferred unless the partial draft has structural problems that surgery cannot fix. The mid-session checkpoint convention (mid-session handoff entry after every batch of ~10–15 ADRs) supports path (a) recovery.
