- Recovery from partial implementation: the most likely partial-implementation failure mode is conversation-context exhaustion during the Architect-role conversation before all ten sections are drafted. Recovery has two paths: **(a) resume in a new conversation** — the new `/start-working` invocation will see `_context/10_non_functional_requirements.md` already partially present (containing the structural preamble plus N of 10 sections drafted), run the framework validator to confirm the file's existence and identify which sections still contain unresolved placeholders, read the mid-session handoff entry in `_context/handoff_log.md` to inherit context, and continue the section sequence from the next undrafted section per CLAUDE.md's per-file resume protocol. The packet's `Approved for apply` flag (set by the gatekeeper-review conversation before the Architect's first conversation began) remains set, so the resume conversation can proceed directly to Apply Mode without re-litigating gatekeeper approval (the resume is a continuation of the same T-004 task, not a new task); **(b) full restart** — `git restore _context/10_non_functional_requirements.md`, abandon the partial sections, restart fresh in a new conversation, possibly with revised section-ordering or detail-level choices if the partial draft revealed that the original choices were ill-calibrated. Path (a) is preferred unless the partial draft has structural problems that surgery cannot fix. The mid-session checkpoint convention from T-002 / T-003 (mid-session handoff entry after every ~3–4 sections) supports path (a) recovery.
