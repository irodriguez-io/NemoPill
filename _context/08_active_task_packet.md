## Task Objective

- **Problem to solve now:** T-009 delivered the **notification leg** of the M-002 walking skeleton — a fired demo alarm renders an on-time Reminder notification with two inline `FLAG_IMMUTABLE` actions ("Take" / "Skip"), but the action handler `ConfirmFromNotificationReceiver` is only a **logging-and-dismiss stub** (ADR-095): the tap is recorded to logcat and the notification is dismissed, but **nothing is persisted**. T-010 takes the **next leg**: turning that tap into a **durably recorded `Confirmation`**. This is M-002 Done When item (4) ("Tapping the inline action invokes a `BroadcastReceiver` (manifest-declared, `exported="false"`, all `PendingIntent`s with `FLAG_IMMUTABLE`), which dispatches to a Confirmation use case, which writes a `Confirmation` row to Room"). It stands up the project's **first Room database** (`NemoPillDatabase`, the single-instance design that file 04 § Architecture Risks makes load-bearing for the cross-module Dose-aggregate transaction) and the **first `:adherence-tracking` production code**, and it realizes the persistence half of `_context/06 § F-006` (steps 1–2, 4, 6: typed parse → dispatch → `Confirmation` write → dismiss). It also forces a decision on the **second load-bearing module seam** of the project: how the `:notifications`-resident receiver reaches `:adherence-tracking`'s use case without an illegal cross-feature Gradle dependency (the first seam — `:scheduling` → `:notifications` via the `:core` event bus — was proven by T-009).
- **Why now:** the notification leg (T-009) is the prerequisite and is merged. The natural next cut is making the tap durable. Slicing the confirmation/persistence leg on its own — **writing** the `Confirmation` row and **dismissing** the notification, but leaving the on-screen Adherence counter (item 5) and the `Dose.status` co-update (no persisted `Dose` exists in M-002) to later slices — keeps a coherent, reviewable spine for the gatekeeper. Per `_context/07 § Task Slicing Rules`, items (4) and (5) are separated so each fits one approval cycle. T-010 is also the **mandatory trigger** for ADR-049 rule (ii) and the ADR-087 decision, because `Confirmation` is the project's first `Confidential`-class Domain `data class` (`_context/13 § Data Classification` row "Confirmation states `taken` / `skipped`") — the exact "first sensitive Domain `data class`" target the T-008 and T-009 close handoffs named.
- **Expected user-visible / system-visible outcome:** After T-010 lands and merges, launching the debug APK, scheduling the demo Reminder, and waiting for it to fire produces the same notification as T-009; **tapping "Take" or "Skip" now writes a `Confirmation` row to `NemoPillDatabase` and dismisses the notification** (instead of only logging). ### Architect Decision Required — the receiver→use-case module seam (surfaced `_context/04` ↔ `_context/06`/`_context/13` conflict)

- **The conflict (do not silently resolve — CLAUDE.md Stop Condition "Multiple framework files disagree about a fact"):** `_context/04 § Gradle dependency graph` (line 170) is unambiguous — "Feature modules **do not** depend on each other … Cross-Bounded-Context communication goes only through `DomainEventPublisher` (in `:core`). `:app` is the only module allowed to depend on the four feature modules." But `_context/06 § F-006` step 2 and `_context/13`'s "PendingIntent → notification-action receiver" row place `ConfirmFromNotificationReceiver` in `:notifications` and have it **directly "invoke `ConfirmDoseFromReminderUseCase` in `:adherence-tracking::application`"** and "resolve DoseId via repository." A class in `:notifications` cannot reference a type in `:adherence-tracking` without an **illegal cross-feature Gradle dependency**. The same tension appears inside file 04 itself (lines 31–32 narrate notifications' PendingIntents "reaching adherence-tracking's confirm path" while line 170 forbids the import). This seam is the second load-bearing module boundary of the project (the first — `:scheduling` → `:notifications` — was resolved via the `:core` event bus in T-009).
- **Candidate resolutions:**
  - **(A) Relocate the receiver to `:app`.** `:app` may depend on both `:notifications` (for the action constants) and `:adherence-tracking` (for the use case). Cost: contradicts file 06/13's explicit placement of the receiver in `:notifications::infrastructure`; moves an OS-ingress class out of the BC that owns the notification surface.
  - **(B) Event-bus listener (mirror T-009).** The receiver publishes a typed signal on the `:core` `DomainEventPublisher`; an `:adherence-tracking` listener invokes the use case. Cost: a notification tap is a **command**, not a past-tense domain **event** — file 02's event catalog (`DoseConfirmed` etc.) is all facts-that-happened; pushing a command through the event bus conflates the two, and adds an async hop on a user-initiated confirm.
  - **(C) ✅ Recommended — a `:core` confirm port bound in `:app`.** Define a neutral contract in `:core` (e.g., `io.nemopill.core.confirm.NotificationConfirmationGateway { suspend fun confirm(doseId: DoseId, status: ConfirmationStatus): Result<Unit, Result.Err> }`). `:notifications`' receiver depends only on the `:core` port (it already depends on `:core`); `:adherence-tracking` provides the implementation that delegates to `ConfirmDoseFromReminderUseCase`; `:app` binds port → impl in DI. This **respects the strict graph** (both feature modules depend only on `:core`), **preserves the command semantics** of F-006 ("invokes the use case"), avoids polluting the domain-event bus, and is **directly precedented by `DomainEventPublisher` itself living in `:core`** as the cross-cutting contract. `ConfirmationStatus` must then also be visible to `:notifications` for the typed parse — declare it (or a `:core` mirror) where both can see it; the Developer confirms the minimal placement (likely `ConfirmationStatus` in `:core` since the IPC `status` enum is genuinely shared-kernel, with `:adherence-tracking::domain` re-using it).
- **PM/Architect recommendation:** adopt **(C)**. **Mandatory:** the Developer confirms the working DI/visibility pattern empirically at execution time (T-009 showed these wiring choices need confirmation against the Hilt/Dagger/Kotlin-metadata reality — ADR-091/ADR-092) and **records the chosen seam as an ADR `Status: Proposed`**, naming this surfaced file-04/06/13 tension and why (C) reconciles it. Security should weigh in during review since the seam touches the IPC-ingress trust boundary (`_context/13`). If the Developer finds (C) infeasible, fall back to (A) and record why — **but do not adopt a seam that imports one feature module into another.**

- **Risks:**
  - **First Room database / schema-export plumbing.** `exportSchema = true` requires the schema-location arg; a missing arg fails the build or silently skips export. The `annotationProcessor` → `ksp` switch for Room is mandatory in this KSP project. *Mitigation:* set the Room KSP schema arg on the `@Database`-hosting module (`:app`); commit `1.json`; the Developer confirms minimal KSP placement and records an ADR (mirror ADR-083/ADR-091).
  - **Room generated `*_Impl` coverage dilution.** Room generates `ConfirmationDao_Impl` / `NemoPillDatabase_Impl`, which would drop the `:adherence-tracking` Kover gate (the same failure mode ADR-091 fixed for Dagger). *Mitigation:* extend the ADR-086 report-level filter to exclude Room-generated classes (`*_Impl`) and `@DaggerGenerated`.
  - **`goAsync()` lifetime vs. the Room write.** The confirm receiver now does real suspend I/O; it must finish well under the OS ~10-second receiver lifetime (`_context/06 § Timeout rule`, exact bound in `_context/12`). *Mitigation:* bound the publish/write coroutine to a short timeout; `finish()` the `PendingResult` in a `finally`; on overrun log `Result.Err.Unexpected` and leave the notification undismissed (the Patient retries).
  - **ADR-049 rule (ii) over-/under-scoping.** A rule that matches every `:core`/Domain `data class` would force redaction on `Result.Ok`/`ReminderFired` (noise); one that matches too narrowly would miss a future sensitive entity. *Mitigation:* the ADR-087 resolution scopes rule (ii) to `Confidential`-data-bearing types (per `_context/13`) via an explicit allow-list or `@NoSensitiveData` marker; the negative test pins both the pass (`Confirmation`/`DoseConfirmed`) and the exempt (`Result.Ok`) cases.
