# Threat Model And Data Classification

## How To Use This File

- This file gives the Security agent something concrete to review against. Without it, security review becomes vibes.
- Capture realistic threats, not exhaustive ones. Focus on what would actually hurt this product.
- Keep this file aligned with `05_engineering_quality_security_and_compliance.md` and `06_integrations_data_flow_and_boundaries.md`.
- If the product handles only public, non-sensitive data and has no auth surface, write `Not applicable` with one sentence of justification.

## Scope And Assumptions

- **Systems in scope:** The NemoPill Android application running on a single Patient's device (per ADR-002 single-Patient framing). Surfaces covered: the six Gradle modules `:app`, `:core`, `:medication-management`, `:scheduling`, `:notifications`, `:adherence-tracking` (per ADR-009); the local Room database `NemoPillDatabase` and its file-system representation in the app sandbox (per ADR-015 / ADR-045); `SharedPreferences` storage for biometric-gate state (per `_context/05 § Authentication`); the AndroidManifest permission set (per `_context/05 § OS permissions and IPC boundaries`); the `BroadcastReceiver` ingress including `BootCompletedReceiver` and AlarmManager-fired receivers (per ADR-012); the `PendingIntent` outbound surface used for notification-action intents (per ADR-023); the optional `BiometricPrompt` app-level gate (per ADR-019); the bilingual Privacy Policy artifact at `legal/PRIVACY_POLICY.{en,es}.md` and its APK-bundled `res/raw{,-es}/privacy.md` copy (per ADR-025 / ADR-048); the Google Play Console submission surface — Data safety declaration, App Signing enrollment, versionCode immutability (per ADR-024 / ADR-041 / ADR-046); Google Play Console Android Vitals as the sole post-release evidence stream (per ADR-049 / ADR-067).

- **Systems explicitly out of scope:**
  - **Any server-side surface.** NemoPill has no backend, no API ingress, no API egress, no DNS lookup, no network socket of any kind — the `INTERNET` permission is absent (ADR-021) and a Konsist assertion prevents any network-API import. Threats against a backend are categorically absent.
  - **Multi-user / caregiver accounts.** Single-Patient framing is locked at ADR-002. Threats requiring multi-tenant isolation, role-based authorization, or cross-user data leakage are categorically absent.
  - **HIPAA-covered exchange.** NemoPill is not a covered entity per ADR-002 and the `_context/05 § Compliance Applicability Matrix` HIPAA row. PHI-handling threats are not analyzed.
  - **Pre-ADR-041 cloud-backup window.** Android Auto Backup to Google Drive was theoretically exfiltratable on pre-ADR-041 builds, but no MVP build has been published to date — the window was theoretical, never operational, and is not enumerated as a residual threat.
  - **Nation-state or law-enforcement-grade forensics against a *locked* device.** NemoPill is not calibrated for this adversary class.
  - **Targeted attacker with physical custody, unlock, and unlimited time** (border seizure, court-ordered custody-dispute access). NemoPill cannot defend against an adversary who possesses all three; this is an accepted residual.
  - **Compromise of Google Play App Signing or Patient's Google account takeover.** Google's signing infrastructure and account-recovery process are trusted upstream prerequisites.
  - **Malicious insider in the organizational sense.** No organization, no employees, no privileged operators — the single-Patient framing collapses "insider" into TA-1 or TA-2 below.

- **Trust assumptions:**
  - **The Android OS is trusted as a hard prerequisite.** If the OS sandbox, the lock-screen subsystem, `BiometricPrompt`, `AlarmManager`, the Android Keystore, or Android Auto Backup honoring `dataExtractionRules` is compromised — by root, by an OS-vendor backdoor, by a malicious Google Play Services update, by a kernel exploit — NemoPill cannot defend itself and we accept that residual. This is a deliberate trade because the entire architecture rests on OS-sandbox protection (per ADR-021 / ADR-041) rather than on application-level cryptography. **This assumption must be reflected in the Patient-facing Terms of Service document when that document is authored** (see §9 Open Security Questions — ToS does not yet exist in `_context/` and is flagged for drafting alongside the Privacy Policy translation deferred to M-006 per ADR-061).
  - **The Patient is trusted with their own data.** Single-Patient framing per ADR-002; there is no malicious-Patient threat model.
  - **The Android OS lock screen is trusted as the primary authentication boundary** (per `_context/05 § Authentication` / ADR-019). When the device is unlocked, the holder is treated as the Patient. The optional app-level `BiometricPrompt` gate is a *defense-in-depth* secondary boundary, default-off, Patient-configurable.
  - **Google Play App Signing is trusted** (per ADR-041). The developer upload key is held on a single dev machine; Google holds the production signing key. Threats requiring forged release signatures are out of scope.
  - **The Gradle dependency graph is trusted under manual review** (per `_context/05 § Dependency and supply chain` / ADR-046). Every new direct dependency and meaningful transitive update is reviewed by the solo developer at commit time; no automated CVE scanning in MVP. A compromised dependency that survives manual review is an in-scope residual (TA-6).
  - **Co-installed third-party apps on the same device are NOT trusted.** `android:exported="false"` on every Activity / Service / BroadcastReceiver except `BootCompletedReceiver` (per `_context/05 § OS permissions and IPC boundaries`), and `PendingIntent.FLAG_IMMUTABLE` on every constructed `PendingIntent` (per ADR-023), close the in-IPC attack surface against TA-5.
  - **The Patient's network is not relied upon.** There is no network surface (per ADR-021); network adversaries (Wi-Fi MitM, captive portal, DNS hijack, hostile ISP) are out of scope by virtue of having no NemoPill traffic to attack.

- **Threat actor profile:** Six actor classes are in scope. The numeric `Capability` rating (1-5) is used in §4 STRIDE scoring.

| Actor ID | Description | Relationship to Patient | Capability (1-5) | Motivation |
| --- | --- | --- | --- | --- |
| `TA-1` | Opportunistic shoulder-surfer or nearby observer | Stranger or acquaintance in physical proximity | 1 | Curiosity, opportunism |
| `TA-2` | Trusted-circle adversary (family member, roommate, caregiver, partner) with intermittent unsupervised access to an *unlocked* device | Trusted-by-Patient relationship | 3 | Curiosity, controlling behavior, custody disputes |
| `TA-3` | Opportunistic thief of an unlocked device | None — adversarial | 2 | Resale value of device; secondary interest in identity data |
| `TA-4` | Forensic recovery off discarded / sold / donated hardware the prior owner did not factory-reset | None — successor possessor | 3 | Opportunistic data recovery; identity-theft secondary market |
| `TA-5` | Co-installed malicious app on the same device | None — same-device tenant | 4 | Data exfiltration, ad fraud, ransomware |
| `TA-6` | Compromised software supply chain — malicious Gradle dependency direct or transitive shipping in a NemoPill release | None — upstream artifact | 5 | Data exfiltration via future code path; sabotage |

## Data Classification

NemoPill's framework-template classes map onto the 4-class sensitivity taxonomy locked in ADR-033 as follows: `Sensitive personal data` → `Confidential`; `App configuration` + `Trigger metadata` → `Internal`; published content contracts → `Public`; `Restricted / Regulated` collapses to `Not applicable` because the same data, held by a HIPAA covered entity, would be regulated — but NemoPill is not a covered entity per ADR-002 and the `_context/05 § Compliance Applicability Matrix` HIPAA row. The regulatory framing is deliberately absent.

| Data Class | Examples | Storage Location | Access Control | Retention | Deletion Trigger |
| --- | --- | --- | --- | --- | --- |

## Trust Boundaries

Boundaries are ordered by where data or execution flow first enters NemoPill's trust envelope — build time → install time → session time → app-open time → runtime events → runtime actions → parallel content lane. The `Trust Score` column scores boundary integrity on 1-5 (5 = OS-enforced and Konsist-asserted, 1 = manual review only). §4 STRIDE references these scores when computing residual risk.

## Identity And Access Model

This section is the threat-model-side restatement of `_context/05 § Authentication` — it captures how identity flows through the trust boundaries enumerated in §3. The substantive authentication policy lives in file 05; this section is the security-review surface.

- **Identity provider:** `Not applicable` at the application level — NemoPill issues no application-level identities per ADR-002 (single-Patient framing). Two **OS-level** subsystems function as de facto identity providers from NemoPill's threat-model perspective: (a) the Android OS lock-screen credential subsystem (PIN, password, pattern, or biometric — Patient-enrolled at the OS level), and (b) the Android Keystore-backed `androidx.biometric.BiometricPrompt` subsystem when the optional app-level gate is enabled per ADR-019. Both are outside NemoPill code; NemoPill consumes their attested success/failure outcomes but does not manage credential material.

- **Authentication factors required:** Two factors, the second being optional and Patient-controlled. **(a) Primary, always applies:** Android OS lock-screen credential — required at the device level, outside NemoPill's control (the device-unlock boundary per §3 Trust Score 4). **(b) Secondary, optional, Patient-configurable:** `BiometricPrompt` app-level gate per ADR-019 — biometric (fingerprint or face) or device-credential fallback (PIN / password / pattern); **default off**; re-engages after Patient-configurable inactivity timeout (1 / 5 / 15 / 60 minutes / Never per ADR-019). When the gate is disabled, factor (a) is the only authentication; when enabled, factor (b) gates the NemoPill data surface specifically. **No application-issued credentials, no application-issued passwords, no MFA second factor in the server sense, no SSO, no passkey, no mTLS, no API key** — per ADR-002 / ADR-021 there is no application surface that would consume any of these.

- **Authorization model:** **Single-Patient implicit authorization.** Every Patient-data row in the local Room database is accessible to the Patient — the only application user — through the in-app UI, with no per-resource access control, no role-based authorization, no attribute-based authorization, no scopes, no policy engine. Per ADR-002's single-Patient framing there is no other principal whose access could be denied. The `BiometricPrompt` gate state per ADR-019 governs whether the data surface is *currently rendered to the screen*, not whether the Patient *is authorized* to read the data — Patient authorization is an architectural invariant, not a runtime decision.

- **Session and token policy:** **No application-issued sessions, no application-issued tokens.** Application session state is bounded by (a) the Android Activity lifecycle (foreground / background transitions managed by the OS — when NemoPill enters background, the gate's inactivity timer starts), and (b) the `BiometricPrompt` gate's inactivity timeout per ADR-019 when the gate is enabled. Server-side session semantics (opaque session IDs, refresh tokens, token rotation, JWT verification, session revocation, device binding, replay protection) do not apply — NemoPill has no token-issuing surface and no remote authentication peer per ADR-021. The biometric-gate enabled flag and inactivity-timeout configuration are persisted in `SharedPreferences` per §2 Data Classification → Internal row; they are configuration state, not session state.

- **Privileged access procedure:** No in-app privileged access — there is no admin role, no developer-mode toggle, no support-impersonation path, no break-glass within a running NemoPill instance. The only privileged actor in the NemoPill operating model is **the solo developer (Isidro Rodriguez)** per ADR-035, whose privileges are scoped to two out-of-band surfaces: (a) **GitHub repository administration** per ADR-026 (commit, branch, tag, release at `https://github.com/izirodriguez/NemoPill`), and (b) **Google Play Console operations** per ADR-046 (upload-key custody on the dev machine, Internal-Testing and Production track promote actions, Data safety form, country availability, App Signing enrollment per ADR-041). Both surfaces are **out-of-band from any running NemoPill instance on a Patient device** — they cannot be invoked from inside the app or by any Patient action. **Privileged-action audit trail:** the git history at the GitHub repo per ADR-026 captures all `legal/`, `_source/`, `_context/`, and `_build/` changes; the Play Console submission history is mirrored back to `_build/play-store-state.md` per ADR-047 and version-controlled in the same repo. There is no in-application logging of privileged actions because no such actions exist inside the running app — the audit boundary coincides with the developer's commit + Play Console actions, both already covered by `_context/05 § Evidence And Audit Trail` per ADR-026.

## Logging And Audit Requirements

NemoPill has **two distinct log surfaces**, both narrow and explicitly scoped: (i) **operational logging** for engineer debugging of unexpected failures — `Log.e` / `Log.w` of `Result.Err.Unexpected` per ADR-067, locally to logcat, with a remote tail to Google Play Console Android Vitals per ADR-049; (ii) **security audit logging** — **does not exist**. The absence of (ii) is an affirmative architectural commitment rooted in ADR-018 (observability is intentionally minimal), ADR-021 (no `INTERNET` permission — no remote log stream possible), ADR-031 (no Patient data in failure evidence — would constrain any future security log anyway), ADR-002 (single-Patient framing — no audit-trail counter-party), and ADR-004 (Confirmation model explicitly has no audit trail in MVP). Security review reads this section asking "where's the auth log? the deletion log? the data-export log?" — the answer for each event class is below.

- **Security-relevant events that must be logged:** `Not applicable` — NemoPill maintains no security audit log. Each framework-template event class resolves as follows:

  - **Auth attempts (`BiometricPrompt` success / failure / cancel):** Not logged. `BiometricPrompt` is OS-managed; success / failure outcomes are consumed by `:app::presentation::BiometricGateScreen` as branch decisions, not log events. No NemoPill code logs auth-attempt counts, timestamps, or biometric-class metadata. **Justification:** ADR-018 + ADR-067 — security observability is unnecessary for a single-Patient device-locked app where the Patient *is* the only valid principal; logging Patient auth attempts adds no defender signal and risks adding Patient-data surface to logcat in violation of ADR-031.

  - **Role changes:** `Not applicable` — no roles exist per §6 (single-Patient implicit authorization). There is no role transition to log.

  - **Data exports:** `Not applicable` — no export feature exists per ADR-022 (no export feature in MVP) and `_context/05 § Data protection` sub-rule (c). The only egress paths for Patient data are the **Patient-initiated hard delete** per ADR-022 / ADR-030 and **uninstall** per ADR-034 — both destroy the data rather than transmit it; neither is logged because the durable evidence is the *absence* of data, not a log entry.

  - **Admin actions:** `Not applicable` — no in-app admin role per §6. The developer's out-of-band privileged actions (GitHub commits per ADR-026, Play Console operations per ADR-046) are audited by `git log` and `_build/play-store-state.md` per ADR-047 — covered in `_context/05 § Evidence And Audit Trail` and §6 above; not duplicated in any in-app log.

  - **Permission revocation events** (`POST_NOTIFICATIONS`, `SCHEDULE_EXACT_ALARM`): not logged in the audit sense. Surfaced to the in-app UI as `Result.Err.NotificationsPermissionRevoked` / `Result.Err.ExactAlarmPermissionRevoked` per ADR-029 — these are *Patient-facing error states*, not security log entries. No timestamp / actor / source-of-revocation is captured because Android does not expose revocation provenance to apps.

  - **App launches, session starts, biometric-gate timeout firings:** not logged. None of these has security value in NemoPill's threat model — every launch is by the Patient (or by an A-001 / A-006-class adversary indistinguishably, which is the very gap the §5 gate-enable recommendation addresses, not a logging gap).

  - **Patient-initiated hard-delete invocation:** not logged in-app. The deletion-event is durable in its own right (the data is gone; uninstall ≡ total deletion per ADR-034); a log entry recording "Patient hard-deleted at time X" would itself be Patient data that the deletion was supposed to remove. Logging the deletion would defeat its purpose.

  - **`Result.Err.Unexpected` failures (operational, not security):** logged via `Log.e` / `Log.w` per ADR-067 as **operational** evidence — not security audit. Content is redacted per ADR-031 (Konsist-asserted via ADR-049's two rules: no string templates in `throw` messages; data class `toString()` redaction). Failures name a `Result.Err` sub-type and an aggregate ID, never a drug name / Dose Schedule / Confirmation history. The remote tail is Google Play Console Android Vitals per ADR-049.

- **Log retention:** Two retention regimes, both narrow.

  - **On-device logcat** (operational log surface): OS-managed circular buffer; retention is hours to a few days depending on device buffer size and log volume; not durable. The Patient cannot view logcat without `adb` + developer options enabled — this is a feature, not a gap, given ADR-031's Patient-data-in-evidence prohibition.

  - **Google Play Console Android Vitals** (operational log surface, remote): Google-managed retention per Play Console policy; the developer (Isidro) reads this via the Play Console UI per ADR-049. Crashes and ANRs are retained per Google's published retention; the in-repo `_build/play-store-state.md` mirror per ADR-047 captures any retention-policy-relevant snapshot taken at submission time.

  - **Security audit log:** `Not applicable` — does not exist per the bullet above; retention is therefore vacuous.

- **Tamper-evidence:** `Not applicable` for security logs (none exist to tamper). For the **operational log surface** that does exist, tamper-evidence is structural rather than cryptographic:
  - (i) The **code that produces operational logs** is version-controlled in the GitHub repo per ADR-026 — any change to logging behavior is captured in `git log` and reviewed under `_context/05 § Quality Gates`.
  - (ii) **Konsist assertions** per ADR-049 verify that no `throw` message uses string templates (preventing accidental Patient-data leakage to crash reports) and that data class `toString()` overrides redact sensitive fields — these are merge-blocking per ADR-044, so tampering with redaction logic fails CI.
  - (iii) **Android Vitals** is Google-managed and outside NemoPill's tamper-evidence boundary — if Google's retention or display were ever compromised, NemoPill has no compensating control and explicitly accepts that residual under §1 Trust Assumption (i) (OS / Google Play infrastructure trusted as upstream prerequisite).

- **Audit access procedure:** `Not applicable` for security logs (none exist). For **operational logs** and **privileged-action audit trail**, the access regime is:
  - **`git log` at `https://github.com/izirodriguez/NemoPill`** per ADR-026 — read access is public (the repository is public); the audit trail of every `_context/`, `_source/`, `_build/`, and `legal/` change lives here.
  - **Google Play Console submission history** — read access is scoped to the Google account holding upload privileges (the solo developer per ADR-041); the in-repo mirror at `_build/play-store-state.md` per ADR-047 makes the substantive submission state inspectable from the public repo.
  - **Play Console Android Vitals view** — read by the solo developer post-release per ADR-049; the data shown there is operational (crash signatures, ANR rates, performance metrics) not security audit; Patients have no view into Android Vitals data about their own device beyond the indirect signal that a crash occurred.
  - **No NemoPill component reads, writes, or transmits a security audit log; no such log exists to read.** This is the affirmative commitment §7 records.

## Open Security Questions

Four open security questions arise from the threat-model analysis above. Q1 is the **load-bearing T-005 agenda item** — the ADR-043 ratify-or-supersede re-evaluation queued from the T-003-close handoff entry — and is the most substantive. ### Q1 — ADR-043: Plain-text Room encryption — ratify, supersede, or partial?

**Context.** Per `THR-010` in §4, the Information Disclosure surface of plain-text Room storage scores Likelihood `2` × Impact `4` = `medium` residual against TA-4 (forensic recovery off discarded hardware). ADR-043 explicitly queued itself for re-evaluation at T-005 against the file-13 adversary enumeration. Three outcomes are permitted by the T-005 packet.

**Option A — Ratify ADR-043 (keep plain-text Room).**
- *Pros:* No dependency added (SQLCipher is ~400 KB compiled + adds cold-start cost incompatible with `_context/10 § Performance Targets` ADR-066 p95 ≤ 2 s NFR without re-measurement); no passphrase-management complexity; no Android Keystore brittleness exposure; ADR-008 reliability-as-architectural-tiebreaker not perturbed; M-001 / M-003 schema work unblocked.
- *Cons:* TA-4 residual stays `medium`; ADR-043's "device-locked threat model" framing rests entirely on the OS sandbox (no second cryptographic layer); if a future Patient-data-richer feature lands (free-text notes, photo attachments), the cost of retrofitting encryption is higher.
- *Residual after:* `medium` (THR-010 unchanged).

**Option B — Supersede with SQLCipher full-database encryption (Proposed ADR).**
- *Pros:* TA-4 residual drops from `medium` to `low` (Impact reduces 4 → 1-2 if cipher unbroken); defense-in-depth second cryptographic layer behind the OS sandbox; aligns with industry norms for medication-related apps.
- *Cons:* SQLCipher dependency (~400 KB compiled) — expands the TA-6 supply-chain surface per `THR-005` / `THR-012`; passphrase management requires Android Keystore-backed master key (introduces Keystore brittleness — Keystore key loss on OS factory reset is permanent, can lock the Patient out of their own data); cold-start performance hit must be measured against the `_context/10` ADR-066 p95 ≤ 2 s NFR before commitment; migration cost for any existing Patient data — currently zero (no MVP installs) but the Room migration discipline per ADR-045 must handle it; cipher-correctness depends on a third-party library inside the trust envelope (small but real TA-6 surface).
- *Residual after:* TA-4 disclosure drops to `low`; new residuals introduced around Keystore brittleness (medium impact, low likelihood) and supply-chain enlargement (folded into existing `THR-005` / `THR-012`).

**Option C — Partial column encryption (e.g., medication-name only) (Proposed ADR).**
- *Pros:* Mitigates the single most-identifying field; lower implementation cost than full SQLCipher; no full-DB cold-start hit.
- *Cons:* Adherence history reveals dosing patterns even with medication identity hidden — TA-4 can correlate timing patterns with publicly-documented dose schedules ("every 8 hours" with a specific time-of-day distribution is recognizable as a specific drug class); doesn't substantively reduce TA-4 residual; introduces per-column encryption complexity in `:medication-management` for marginal benefit.
- *Residual after:* `medium` (similar to Option A — medication identity hidden but pattern still observable).

**Recommendation: Option A — Ratify ADR-043.** The recommendation rests on four observations from the file-13 analysis:

1. **TA-4 Likelihood is genuinely `2`, not higher.** Modern Android factory-reset on stock devices reliably wipes app data; chip-off forensics requires a determined adversary willing to invest in commodity hardware and time, rare for the medication-adjacent data class (no payment-card resale value, no identity-theft credential value).
2. **The cheapest residual reduction is on the *Patient-action* side, not the *cryptographic* side.** A mandatory pre-discard checklist in the upcoming ToS (per the §5 A-002 Response column) addresses the root cause — discarded hardware retaining data because the prior owner didn't wipe — at far lower cost than SQLCipher integration. This investment also has spillover value: it primes Patient mental models around the OS-trust-as-hard-prerequisite assumption from §1.
3. **Option B introduces new residuals that may not net positive.** Android Keystore brittleness is a real failure mode (Keystore key loss on OS factory reset is permanent and irrecoverable), and the SQLCipher dependency expands the TA-6 supply-chain surface that's already the largest single open vulnerability (THR-005 / THR-012 + A-005 all stem from supply-chain fragility). Trading a `medium`-residual TA-4 surface for new `medium`-residual Keystore + supply-chain surfaces is not obviously a security improvement.
4. **Option C doesn't help.** Adherence patterns are themselves disclosive even with medication names redacted — the dosing rhythm of a typical antipsychotic, anticonvulsant, or anti-hypertensive regimen is identifiable from timing alone to anyone with basic clinical-pharmacology knowledge.

**Compensating controls (mandatory if Option A is accepted):**

- **(i) Pre-discard checklist becomes a mandatory ToS content requirement** — the upcoming ToS (see Q2 below) must include explicit Patient guidance: (1) invoke Settings → "Delete all my data" per ADR-022 / ADR-030; (2) uninstall NemoPill per ADR-034; (3) perform Android factory reset — in that order, before discarding the device.
- **(ii) Revisit triggers for encrypted-Room** — Option B becomes the default recommendation if any of the following triggers fires: (a) a Patient-data-richer feature lands (free-text notes, photo attachments, Patient-name capture, contact-info capture); (b) the regulatory bar rises (Chile's pending GDPR-style law replacement activates encryption-at-rest expectations, Brazil LGPD re-entry, CCPA / CPRA reasonable-security-measures interpretation tightens); (c) a post-launch incident in the TA-4 class occurs (Patient-reported data-recovery breach via GitHub Issues per ADR-061; security-researcher disclosure). When any trigger fires, a Proposed ADR superseding ADR-043 with Option B is the standard response.
- **(iii) §1 Trust Assumption (i) is load-bearing for this decision** — the OS-trust-as-hard-prerequisite assumption is acknowledged as load-bearing in §1 already; the ratification of ADR-043 is conditional on that assumption holding.

**Outcome of this re-evaluation:** ADR-043 is ratified by this analysis. **No new ADR lands for the encryption decision itself** per Option (a) of the T-005 packet's permitted outcomes. The compensating controls are enforced through Q2 (ToS authoring) and Q3 (biometric-gate revisit) below, and the ratification event itself is recorded in `_context/09_decision_log.md` as a new ADR alongside the device-locked threat envelope ADR.

### Q2 — Terms of Service authoring

**Context.** NemoPill does not currently have a Patient-facing Terms of Service document. The bilingual Privacy Policy at `legal/PRIVACY_POLICY.{en,es}.md` per ADR-025 / ADR-048 (with Spanish long-form translation deferred to M-006 per ADR-061) is the only published content contract. The threat-model analysis in §§1, 3, 5, and Q1 above surfaced multiple guidance items that **must** reach the Patient via a Patient-facing legal artifact — and the Privacy Policy is the wrong surface for them (it describes how data is handled, not what the Patient must do).

**Deferred to:** the same future task that completes the Spanish Privacy Policy translation per ADR-061 (currently planned for M-006 pre-launch). The ToS is drafted alongside the Privacy Policy translation; both ship as a coordinated bilingual legal artifact set in `legal/PRIVACY_POLICY.{en,es}.md` + `legal/TOS.{en,es}.md` (path convention to be confirmed by the future task).

**ToS content requirements established by file 13** — the future ToS-drafting task must cover all of these:

- **(a) OS-trust-as-hard-prerequisite acknowledgment** per §1 Trust Assumption (i) — the Patient is informed that if the Android OS, lock screen, biometric subsystem, AlarmManager, Keystore, or Auto Backup `dataExtractionRules`-honoring subsystem is compromised by root, OS-vendor backdoor, malicious Google Play Services update, or kernel exploit, NemoPill cannot defend itself and accepts that residual; the Patient is therefore responsible for keeping their device OS up to date and avoiding sources that compromise OS integrity.
- **(b) Pre-discard checklist** per §5 A-002 Response and Q1 Compensating control (i) — (1) invoke Settings → "Delete all my data"; (2) uninstall NemoPill; (3) perform Android factory reset — mandatory Patient guidance before discarding the device.
- **(c) Lock-screen sensitive-content recommendation** per §5 A-003 Response — the Android Settings → Notifications → "Show sensitive content on lock screen: Off" toggle is the strongest mitigation against shoulder-surfing; NemoPill cannot enforce this from app code (it's an OS-level setting).
- **(d) Biometric-gate enable + audit-enrolled-biometrics recommendations** per §5 A-001 / A-004 / A-006 — the Patient is informed that (i) the app-level `BiometricPrompt` gate is default off and Patient-configurable per ADR-019; enabling it is the single most effective in-product mitigation against TA-2 and TA-3; (ii) `BiometricPrompt` accepts any device-enrolled biometric by Android API contract — Patients in shared-device relationships should periodically audit OS-enrolled biometrics via Android Settings → Security → Fingerprint / Face Unlock and remove unrecognized enrollments.

The ToS is the **only Patient-facing legal artifact** in which these guidance items make sense — Privacy Policy is the wrong surface (describes processing, not behavior); UI tooltips are too transient; release notes per ADR-047 are the wrong audience.

### Q3 — Biometric-gate default-on revisit

**Context.** ADR-019 fixed the `BiometricPrompt` app-level gate's default state as **off**, with Patient-configurable enable + inactivity timeout. The §4 STRIDE analysis surfaced that this default state is responsible for the **two highest residual scores in the table** — `THR-001` (Spoofing via device unlock) and `THR-009` (Information Disclosure via trusted-circle UI access) both score `high` residual when gate is disabled and `medium` when enabled. Flipping the default from `off` to `on` would drop both residuals from `high` to `medium` in one decision.

**Open question:** Should ADR-019 be superseded with `BiometricPrompt` default-on?

**Trade-off surface (not analyzed in this file — deferred to a future task that owns the UX cost):**

- *Pros:* Drops `THR-001` and `THR-009` from `high` to `medium`; aligns the default with the threat model's strongest in-product mitigation; addresses the most actionable finding from the §4 analysis.
- *Cons:* Patient onboarding friction (default-on gate adds a step to first-launch); some Patients may find the BiometricPrompt friction discouraging and uninstall (NemoPill's primary failure mode per `_context/03 § Patient persona`); ADR-019 chose default-off based on UX trade-offs that this analysis does not re-litigate.

**Deferred to:** a future task that owns the UX side of the trade-off — most likely M-006 pre-launch when the Play Console listing copy + onboarding flow are being finalized. The task packet for that future task should explicitly cite this open question. If the future task decides to flip the default, a Proposed ADR superseding ADR-019 lands at `Status: Proposed` per the post-T-003 supersession convention.

### Q4 — Automated CVE scanning activation

**Context.** Per `_context/05 § Dependency and supply chain` and `_context/05 § Notes on what's deferred out of Security Guardrails`, automated CVE scanning (OWASP dependency-check or equivalent) is explicitly deferred in MVP. The §4 STRIDE analysis showed that `THR-005` (TA-6 in-process tampering) and `THR-012` (TA-6 exfiltration) both score `medium` residual largely because **manual review at commit time** is the only gate against compromised supply-chain dependencies — and `A-005` in §5 narrates this attack path step by step. The Gradle dependency boundary in §3 was rated the weakest in the table at Trust Score `2`.

**Open question:** When should automated CVE scanning be activated?

**Activation triggers (not analyzed in this file — deferred to a future task):**

- *Threshold trigger:* the dependency graph (direct + meaningful transitive) crosses a complexity threshold where manual review becomes unreliable. Indicative thresholds: ≥ 30 direct deps; ≥ 5 transitive deps per direct dep on average; any dep introducing native code (`.so` / JNI) that the developer cannot easily audit. The exact threshold is the future task's decision.
- *Regulatory trigger:* any Compliance Applicability Matrix row transitions from `Not applicable` or `Planned` to `Applicable` and the activated standard requires CVE-scanning evidence — e.g., a future SOC 2 / ISO 27001 commitment (currently `Not applicable` per `_context/05 § Compliance Applicability Matrix`).
- *Incident trigger:* any TA-6-class incident (Patient-reported strange behavior via GitHub Issues per ADR-061 traced to a dep, upstream CVE disclosure for a dep already in `gradle/libs.versions.toml`, or in-the-wild crash anomaly per ADR-049 traced to a dep) automatically activates CVE scanning as part of the recovery.

**Deferred to:** the first future task where any of the triggers fires. The activation lands a new Konsist or Gradle-task ADR (e.g., "OWASP dependency-check wired as merge-blocking per ADR-044's CI conventions") in `_context/09_decision_log.md`.
