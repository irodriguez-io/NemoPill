# Integrations, Data Flow, And Boundaries

## How To Fill This File

- Document every external dependency and every important data handoff.
- Focus on where data crosses boundaries, who validates it, and how failures are handled.
- Keep this file synchronized with the architecture and compliance rules when contracts change.

## Integrations Register

NemoPill declares no external SaaS, backend, or network integrations by design (file 05 § Data protection: no `android.permission.INTERNET`). The "integrations" enumerated below are the Android OS subsystems and the build-time Google Play surface the app collaborates with. Every row maps to one or more business rules from `_context/02_domain_model_and_business_rules.md` or compliance obligations from `_context/05_engineering_quality_security_and_compliance.md`.

## Data Flow Narratives

This section enumerates every workflow that crosses a boundary listed in the Integrations Register or the Trust Boundaries section below. Each row's *Processing Summary* is a numbered sequence of steps anchored to the BR rules from `_context/02_domain_model_and_business_rules.md`, the UC IDs from `_context/03_user_experience_and_use_cases.md`, and the modules from `_context/04_solution_architecture.md`. Each row's *Failure Handling* enumerates the failure modes the flow can hit, the recovery path, and the audit / observability artifact the failure produces.

The 15 flows cover every BR-004 / BR-009 / BR-010 / BR-012 OS-crossing path that file 05 requires test coverage for, plus the read-only and the hard-delete paths.

| Flow ID | Trigger | Input Source | Processing Summary | Output Or Side Effect | Failure Handling |
| --- | --- | --- | --- | --- | --- |
| F-001 — Add Medication and bootstrap horizon | Patient submits the Add-Medication form. | Patient (UI form). | (1) `AddMedicationViewModel` performs format-level validation per file 05 § Input validation. (2) `AddMedicationUseCase` (UC-001) in `:medication-management::application` constructs Domain value objects — `Strength`, `Form`, `DateRange`, `Frequency` — enforcing BR-002 / BR-003 at construction. (3) `MedicationRepository.persist(...)` writes the `Medication` + initial `DoseSchedule` inside a single `room.withTransaction { }` block preserving the BR-001 single-active-DoseSchedule invariant. (4) Emits `MedicationCreated` on `DomainEventPublisher`. (5) `:scheduling::application::GenerateDosesUseCase` listener calls `ClockPort.now()` + `TimezonePort.currentZone()`, materializes 3 days of `Dose` rows per BR-009, recording `Dose.generatedInTimezone`. (6) For each Dose, `SchedulerPort.schedule(doseId, scheduledAt)` registers the PendingIntent with AlarmManager — idempotent on `doseId` per BR-004. | Persisted `Medication` + active `DoseSchedule`; persisted `Dose` rows (status = `pending`); AlarmManager registrations for the 3-day horizon. | BR-002 / BR-003 violation → `Result.Err.ValidationFailed` from UC-001, surfaced inline by the ViewModel per file 03 acceptance criteria; no events emitted; Room left untouched. `room.withTransaction { }` failure → full rollback; `Result.Err.Unexpected` logged at `Log.e` with tag `NemoPill.medication.AddMedication` per file 05 § Observability. `SchedulerPort.schedule` `SecurityException` from a revoked `SCHEDULE_EXACT_ALARM` grant → adapter catches and returns `Result.Err.ExactAlarmPermissionRevoked`; Dose persists in Room with status = `pending`; daily `HorizonExtensionWorker` re-attempts registration on next pass; UX surfaces a "permission needed" affordance per file 03. **Audit:** `MedicationCreated` and per-Dose `DoseMaterialized` events are observable through `DomainEventPublisher` per file 05 § Observability. |
| F-002 — Replace Dose Schedule | Patient edits an active Medication's Dose Schedule. | Patient (UI form). | (1) `ReplaceDoseScheduleViewModel` does format-level validation. (2) `ReplaceDoseScheduleUseCase` (UC-002) constructs new `DoseSchedule` value objects (BR-002 enforced). (3) Inside `room.withTransaction { }`: deactivate the prior `DoseSchedule` (BR-001), activate the new one, and call `DoseRepository.deleteFuturePending(medicationId)` per BR-006 forward-only. (4) Emits `DoseScheduleReplaced`. (5) `:scheduling::application::GenerateDosesUseCase` listener: `SchedulerPort.cancel(doseId)` for each deleted future-pending Dose, then re-materializes the horizon under the new `DoseSchedule` per BR-009, and re-registers Reminders. | Updated `DoseSchedule` rows; replaced future-pending `Dose` rows; replaced AlarmManager registrations. | BR-002 violation → `ValidationFailed`, inline error. Transaction failure → rollback, `Unexpected` logged; **no partial state** because BR-001 + BR-006 share the same `withTransaction`. `SchedulerPort.cancel` / `.schedule` `SecurityException` → handled as F-001 step 6. **Audit:** `DoseScheduleReplaced` + per-Dose `DoseMaterialized` events. |
| F-003 — Archive Medication | Patient archives a Medication. | Patient (UI tap with confirmation). | (1) `ArchiveMedicationUseCase` (UC-003). (2) Inside `room.withTransaction { }`: set `Medication.isActive = false`; `DoseRepository.deleteFuturePending(medicationId)` per BR-007. (3) For each deleted future-pending Dose, `SchedulerPort.cancel(doseId)`. (4) Emits `MedicationArchived`. | Updated `Medication` row; deleted future-pending `Dose` rows; canceled AlarmManager registrations. | Re-archiving an already-archived Medication is a **no-op** per file 04 § Repository idempotency. Transaction failure → rollback + `Unexpected`. **Audit:** `MedicationArchived` event. |
| F-004 — Daily horizon extension | `HorizonExtensionWorker` periodic execution (daily). | WorkManager-managed periodic schedule (initial enqueue at first `MedicationCreated`). | (1) Worker reads every active Medication's active `DoseSchedule`. (2) For each, computes the next day's `Dose` set against current `ClockPort.now()` + `TimezonePort.currentZone()`. (3) Bulk insert via `DoseRepository.insertHorizon(...)` — **idempotent on `(medicationId, scheduledAt)`** per BR-009. (4) For each newly inserted Dose: `SchedulerPort.schedule(doseId, scheduledAt)`. (5) Worker returns `Result.success()`. | New `Dose` rows; new AlarmManager registrations. | Idempotent unique-key catches any duplicate that a missed-run / re-run race would have inserted. Worker exception → `Result.failure()` → WorkManager-managed backoff retry. `SchedulerPort.schedule` failure → per-Dose `Unexpected` logged; Dose persists; next-day pass re-attempts registration. **Audit:** per-Dose `DoseMaterialized` events. |
| F-005 — On-time Reminder fire (BR-004) | `AlarmManager` fires the registered `PendingIntent` at `Dose.scheduledAt`. | Android OS (AlarmManager). | (1) Manifest-declared `BroadcastReceiver` in `:scheduling::presentation` receives the Intent and parses the `doseId` extra per file 05 § Input validation entry point. (2) Receiver uses `goAsync()` to bound the coroutine scope and invokes `FireReminderUseCase(doseId)` in `:scheduling::application`. (3) Use case reads the Dose from Room and checks two preconditions: (a) `Dose.status == pending`, and (b) `ClockPort.now() <= scheduledAt + 1h` per BR-010 — together they enforce BR-004 exactly-once and skip-if-status-changed. (4) Emits `ReminderFired(variant = onTime)` on `DomainEventPublisher`. (5) `:notifications::application::ReminderFiredListener` subscribes and invokes `NotificationPort.presentReminder(variant = onTime)`. (6) `NotificationManagerCompat` renders the Reminder with the inline-action `PendingIntent`s wired to `ConfirmFromNotificationReceiver` in `:notifications::infrastructure` per Section 2's PendingIntent contract. | A single user-visible Reminder notification; Dose remains `pending` until F-006 / F-007 confirms it. | Dose already `taken` / `skipped` / `missed` at fire time (raced confirm or post-deadline) → `FireReminderUseCase` returns early, no event emitted, no notification — this is the BR-004 exactly-once guard. `POST_NOTIFICATIONS` revoked → `NotificationPort.presentReminder` returns `Result.Err.NotificationsPermissionRevoked`; Dose stays `pending`; F-008 sweep eventually flips to `missed`. Receiver-scope coroutine timeout (`goAsync()` ANR risk) → bounded to a short timeout; on overrun, `Unexpected` logged; sweep recovers. **Audit:** `ReminderFired` event. |
| F-006 — Confirm from notification (UC-004) | Patient taps "Taken" or "Skipped" on the lock-screen notification action. | Android OS (`PendingIntent` invocation) → `ConfirmFromNotificationReceiver` in `:notifications::infrastructure`. | (1) Receiver parses the `(doseId, status)` extras **at the entry point** per file 05 § Input validation. (2) Receiver invokes `ConfirmDoseFromReminderUseCase(doseId, status)` (UC-004) in `:adherence-tracking::application`. (3) Use case enforces BR-005 (enum bounds — typed at parse), BR-011 (24-hour window — vacuously satisfied at on-time confirm), and the Dose-aggregate transactional invariant. (4) Inside `room.withTransaction { }`: write `Confirmation` (1:1 with Dose, idempotent on `doseId` per file 04 § ConfirmationRepository) **and** update `Dose.status` to `taken` or `skipped` — the single-database / single-transaction design from file 04 § Architecture Risks is what makes this atomic. ## Reliability Rules

NemoPill has **no network boundary**, so most of the framework template's reliability vocabulary (call latency, downstream failure, distributed retry, cross-system reconciliation) maps to `Not applicable`. What remains meaningful and is captured below: OS-callback timeouts, WorkManager retry semantics, the per-port and per-flow idempotency keys that file 04 already locked in, in-process event ordering, and intra-app partial-failure recovery when a multi-port flow can't be wrapped in a single transaction. Each bullet anchors to the specific BR rules from `_context/02_domain_model_and_business_rules.md` and the ports / quality gates from `_context/04_solution_architecture.md` and `_context/05_engineering_quality_security_and_compliance.md` that enforce it.

- **Timeout rule.** Three distinct timeouts apply at three different layers. **(a) OS-callback entry-point timeouts:** every `BroadcastReceiver` (`BootCompletedReceiver`, `TimezoneChangedReceiver`, the AlarmManager Reminder-fire receiver, `ConfirmFromNotificationReceiver`) uses `goAsync()` with a coroutine scope bounded to a short timeout (well under the OS-enforced ~10-second receiver lifetime — exact value locked in `_context/12_environments_and_devops.md`). On overrun the coroutine is canceled, `Result.Err.Unexpected` is logged per file 05 § Observability, and the next periodic sweep / next foreground refresh closes the gap. WorkManager `Worker` runtime is bounded by WorkManager's default 10-minute envelope; both `HorizonExtensionWorker` and `PendingToMissedSweepWorker` are designed to complete well inside that. **(b) Business-rule timeouts:** BR-010's 1-hour late-Reminder window is the timeout that separates "fire late with the late-text variant" from "transition to missed"; BR-011's 24-hour retroactive window is the timeout that separates "retroactive Confirmation allowed" from "Dose locked at missed". These are pure-Domain windows evaluated against `ClockPort.now()` — there is no asynchronous deadline machinery, just an instant-comparison every time a relevant use case runs. **(c) Application use case timeouts:** suspend functions in `:application` have **no internal timeout** — they complete or throw; the caller (ViewModel, Worker, Receiver) owns the upper bound. UI flows do not race a wall-clock — the Patient races a wall-clock, and the Domain-layer windowed rules above are what they race against.

- **Retry rule.** **Retry policy is per-entry-point.** **WorkManager workers** (`HorizonExtensionWorker`, `PendingToMissedSweepWorker`) return `WorkerResult.failure()` on unexpected exception, which triggers WorkManager's default exponential-backoff retry; backoff parameters are pinned in `_context/12_environments_and_devops.md`. **AlarmManager Reminders are not retried by NemoPill** — BR-004 exactly-once is enforced by `SchedulerPort` idempotency on `doseId`, and OS-induced delays (Doze exit, post-boot fire) are handled by the F-009 late-fire path and the F-008 sweep, not by re-registration. **UI use cases never auto-retry** — failures return as `Result.Err.*` to the ViewModel and the Patient re-taps if they choose to retry, per file 03's UX contract; auto-retrying a mutation behind the Patient's back violates the "Patient is the authoritative actor" framing from file 03. **`BootCompletedReceiver` re-registration is idempotent re-attempt on each boot**; if `SchedulerPort.schedule` fails for a Dose because `SCHEDULE_EXACT_ALARM` is revoked, the receiver does not loop inside a single boot — it relies on the next `HorizonExtensionWorker` pass and the next app-foreground UX prompt to recover. **`ConfirmFromNotificationReceiver` does not retry on transaction failure** — the source notification is **not dismissed**, so the Patient can retry by tapping again or from in-app per F-006.

- **Idempotency rule.** Every cross-boundary write the app performs has a documented dedupe key, and every reader / subscriber honors it. **`SchedulerPort.schedule(doseId, scheduledAt)` is idempotent on `doseId`** per BR-004 / file 04 § SchedulerPort — re-registering the same Dose's Reminder is a no-op. **`DoseRepository.insertHorizon(...)` is idempotent on the unique key `(medicationId, scheduledAt)`** per BR-009 — re-running the daily horizon extension cannot create duplicate Doses for the same Medication × scheduled time. **`MedicationRepository.archive(medicationId)` is idempotent** per file 04 § MedicationRepository — re-archiving an already-archived Medication is a no-op. **`ConfirmationRepository` is 1:1 unique on `doseId`** per file 04 § ConfirmationRepository — F-006's confirm-from-notification path is safe against double-tap, and UC-007 corrections use Option A in-place mutation with the same `confirmationId` per file 02. **Domain-event subscribers dedupe on the event's documented key** per file 04 § DomainEventPublisher — every event in file 02 § Domain Events specifies the field subscribers use to detect a replay. **F-014 hard delete is idempotent** — truncating an already-empty Room table is a no-op, and `SchedulerPort.cancelAll()` / `WorkManager.cancelUniqueWork(...)` / `NotificationManagerCompat.cancelAll()` are all safe to call against an empty state.

- **Ordering rule.** The in-process `InProcessEventBus` is a Kotlin Coroutines `MutableSharedFlow<DomainEvent>` per file 04 § DomainEventPublisher; **per-subscriber emission order is preserved** by the coroutines spec — a subscriber receives events in the order they were emitted. A small **replay buffer** accommodates subscriber-late catch-up for boot-ordering cases where subscribers register at app startup but events were emitted by the bootstrap path before registration completed. **Transactional ordering inside `room.withTransaction { }` is the strict atomicity guarantee** — the BR-001 atomic deactivate-old / activate-new, the BR-006 forward-only Dose replacement, the BR-007 archive-with-future-pending-deletion, the F-006 / F-007 Confirmation-plus-Dose-status write, the F-012 timezone-change regeneration, and the F-014 hard delete each run inside one `withTransaction` block per file 04 § Repository design. **F-010 batched-summary aggregation accepts eventual-consistency**: events from a single sweep pass are aggregated into one batched-summary notification, but if `DoseMissedListener` presents the first arrival before the rest arrive, the result is one + (N − 1) silent-missed notifications instead of a single batched-summary. BR-010 documents this as the acceptable degraded path, justified by the cost of a synchronization barrier versus the UX gain. **There is no global event-emission ordering across subscribers** — two subscribers may process the same event in either order, and any cross-subscriber state derivation must dedupe and re-converge rather than depend on inter-subscriber ordering.

- **Partial failure rule.** **The unit of consistency is the `room.withTransaction { }` block.** Inside a single block, Room rolls back atomically — there is no partial-Room state. The non-trivial case is a **multi-port flow** that cannot be wrapped in a single transaction: most often, *Room write succeeded, then `SchedulerPort.schedule(...)` failed* (e.g., F-001 step 6 SecurityException on a revoked `SCHEDULE_EXACT_ALARM` grant). The recovery is the same in every such case, and is the load-bearing reason F-008 (sweep) and F-011 (boot-survival) exist: **(i)** the Room write persists (Dose remains `pending`); **(ii)** the SchedulerPort failure surfaces as `Result.Err.ExactAlarmPermissionRevoked` and is logged per file 05 § Observability; **(iii)** the next `HorizonExtensionWorker` pass re-attempts registration (idempotent on `doseId`); **(iv)** if the permission is still revoked, the `PendingToMissedSweepWorker` eventually flips the Dose to `missed` per BR-010; **(v)** the next app-foreground surface presents a "permission needed" affordance per file 03 acceptance criteria. **`NotificationPort` partial failures** (e.g., `POST_NOTIFICATIONS` revoked) follow the same pattern — the Reminder-fire transaction is complete, the Patient does not see the notification, the sweep handles the eventual transition to `missed`. **F-014 hard delete has no partial-recovery path** — the destructive transaction either succeeds atomically or rolls back atomically; if `SchedulerPort.cancelAll()` / `WorkManager.cancelUniqueWork(...)` fail after Room truncation, the residual is *future alarm fires for now-deleted Doses* and *workers that find an empty database* — both of which are no-ops at fire time (F-005 returns early on missing Dose, sweep finds nothing to sweep). The system is **fail-safe** under this asymmetry: Doses cannot survive a successful Room truncation, only registered alarms can, and an alarm that fires for a deleted Dose dies quietly. **There are no cross-process or cross-system compensating actions** — NemoPill has neither.

## Error Handling And Recovery

NemoPill is a single-process clientside Android app with no service surface, no on-call rotation, and no operator team — the "operator" in the framework template's sense is the solo developer. Error handling is anchored in file 04 § Error handling rule's **two-tier `Result<T, E>` policy** (expected business failures as typed sealed sub-types; unexpected failures translated into `Result.Err.Unexpected` and logged). This section catalogs how each tier surfaces to the Patient, what the developer's recovery toolkit looks like, and which failure-specific artifacts capture each failure shape. The project's **general audit trail** — the seven in-repo artifact classes — lives in `_context/05_engineering_quality_security_and_compliance.md` § Evidence and is not restated here.

- **User-facing failure behavior.** The Patient sees one of two surfaces per file 03's UX contract. **(a) Expected business failures** are returned as typed `Result.Err` sealed sub-types and rendered as **non-technical inline UX copy** at the surface that initiated the action — no logcat noise, no crash, no generic dialog. The catalog accumulated through this file is: `Result.Err.ValidationFailed` (BR-002 invalid date range, BR-003 zero/negative quantity — rendered inline at the form field that failed); `Result.Err.RetroactiveWindowExpired` (BR-011 — the Dose-card action buttons are removed once the window closes per file 01 Q2, so this Err is rarely surfaced as a message because the affordance is gone before the Patient can tap it); `Result.Err.ExactAlarmPermissionRevoked` (F-001 / F-011 — rendered as a "permission needed" affordance per file 03 acceptance criteria, deep-linking to OS Settings); `Result.Err.NotificationsPermissionRevoked` (F-005 — same "permission needed" affordance); `Result.Err.UnknownDose` (F-006 — defensive only; if seen, the notification stays so the Patient can retry from in-app); `Result.Err.UnexpectedNotificationPayload` (F-006 defensive parse — same behavior as `UnknownDose`). **(b) Unexpected failures** are caught at the Application / Infrastructure boundary per file 04 § Error handling rule, translated into `Result.Err.Unexpected`, and rendered as a **generic error message with a retry option** — never with technical details, never with stack traces, never with database state. The Patient is **never alerted by a runtime "something is wrong" channel** per file 05 § Observability — Patient-facing communication about misconfiguration is a product-UX concern, not an observability one.

- **Operator-facing recovery steps.** The "operator" is the solo developer; there is no on-call, no PagerDuty, no runbook execution surface. The developer's **recovery toolkit** for debugging an `Unexpected` failure is: **(a)** `logcat` reads filtered by the `NemoPill.<module>.<class>` tag convention per file 05 § Observability — every `Result.Err.Unexpected` log line names the module, class, and the underlying exception type; **(b)** a **temporary debug-only subscriber** on `DomainEventPublisher` (`InProcessEventBus`) to introspect the live event stream — file 04 § Architecture Risks notes this is a debug technique, not a production capability, and it is added and removed as a transient development artifact, not committed; **(c)** **direct Room database inspection** via Android Studio's Database Inspector (debug builds) or an `adb` pull of the database file (dev builds only) — the database is unencrypted per file 05 § Data protection, which makes this trivially possible on a debug-rooted device but does not weaken the production threat model (file 13); **(d)** the **deferred crash-reporting hook** in file 12 (Crashlytics or equivalent), once landed, surfaces unhandled exceptions and `Unexpected`-flagged failures via the crash-reporter dashboard during pre-release testing and post-release operation. **There are no dashboards, no replay queues, no manual-intervention surfaces**, and no runtime configuration that the developer can change without shipping a new release — for OS-level behavior changes (post-Android-update Doze tuning, post-Play-Services regressions, etc.), the recovery path is **a new release with a fix**, not a remote toggle.

- **Audit or evidence requirements (failure-specific).** The general project audit trail (the seven in-repo artifact classes — `_context/` files, ADRs, handoff log, active-task-packet git history, Privacy Policy versions, Play Console snapshots, dependency-review commit messages) lives in file 05 § Evidence. **This bullet covers only failure-specific evidence.** **(i) Only the `Result.Err.Unexpected` branch is logged**, via `Log.e` (or `Log.w` for recoverable adapter surprises that nonetheless propagate up as `Unexpected`), with the `NemoPill.<module>.<class>` tag per file 05 § Observability. **(ii) Expected business-rule failures are never logged** — `ValidationFailed`, `RetroactiveWindowExpired`, `ExactAlarmPermissionRevoked`, `NotificationsPermissionRevoked`, `UnknownDose`, `UnexpectedNotificationPayload` all return as typed `Result.Err` and surface to UX without producing a log line. **(iii) Domain events emitted from a partial-failure recovery path** (e.g., `DoseMissed` from an F-008 sweep that picks up a Dose left `pending` after a failed F-001 step 6 registration) are the recoverable evidence — they are observable through `DomainEventPublisher` per file 05 § Observability, not separately logged. **(iv) Crash reports**, once the deferred Crashlytics-equivalent hook in file 12 lands, capture the post-release surface of unhandled exceptions and `Unexpected`-flagged failures — and only those; expected business failures still do not produce a crash report. **(v) No Patient data is recorded in failure evidence** — log lines and (forthcoming) crash reports name failures by `Result.Err` sealed sub-type and by aggregate ID (`MedicationId`, `DoseId`, `ConfirmationId`), **never** by drug name, dose schedule, or Confirmation history. This is a hard rule, not a preference: it operationalizes file 05 § Data protection's "no transmission off-device" commitment at the evidence-collection layer, so that a future crash reporter that does transmit to a vendor cannot leak sensitive personal data through error reports.

## Data Ownership And Retention

This section captures who owns each major record type, how long records are retained, and whether they may be exported. The rules below operationalize file 01's scope decisions, file 04's architecture (single on-device Room database, no cloud sync, no replica), and file 05's data-protection commitments (no transmission off-device, Patient-controlled irreversible hard delete, no export feature in MVP).

- **Source of truth by major record type.** Every record NemoPill handles lives **on-device, in the single `NemoPillDatabase` Room instance** owned by `:app::di::AppModule` per file 04 § Local persistence. There is no upstream backend, no cloud sync, no replica, no shadow store — Room is **the** source of truth, and the architecture is what it is precisely so that this can be a true statement. **The asymmetry that matters** is that the Patient is the *upstream authority* and the app is the *durable system of record for the resulting history.* The Patient is the authority on: which Medications they take (file 01's Ubiquitous Language treats Medication as the Patient's identifying handle on the drug, not a clinical record); what Dose Schedule those Medications follow (the schedule is the Patient's chosen rule, not a prescription's authoritative text); and whether each Dose was taken or skipped (Confirmation is, by file 02's domain-language definition, the Patient's recorded response). The app's job is to **persist that authority faithfully** — to never silently transform what the Patient said, to never derive Confirmation state from inference, and to expose corrections (UC-007, Option A in-place mutation per file 02 / file 04) **because the Patient is the authority**, not because the system is unsure. **Per-Bounded-Context ownership** maps to file 04 § Module map: `Medication` and `DoseSchedule` are owned by `:medication-management`; `Dose` is owned by `:scheduling`; `Confirmation` is owned by `:adherence-tracking`. Cross-module reads (e.g., `:adherence-tracking` reading `Dose` rows to compute BR-008 Adherence) go through the read-only paths on the relevant repository per file 04 § Interface And Adapter Strategy, but the *write authority* for each row type stays inside the owning module.

- **Retention or deletion rules.** Three composing rules. **(a) Indefinite on-device retention by default.** No auto-purge, no Dose-history trimming, no Adherence-history rollup that loses precision. BR-008 Adherence is computed on demand over whatever Dose × Confirmation history is on disk; the rolling-window framing (e.g., "30-day Adherence") is a *query parameter*, not a retention boundary — older Doses remain readable for any window the Patient cares to look at. The system has no opinion on how many years of Dose history is too many. **(b) Patient-controlled irreversible hard delete** per file 05 § Data protection (also exercised by F-014). The Settings → "Delete all my data" action drops every Room table, cancels every AlarmManager registration, cancels every WorkManager unique work, dismisses every live notification, and resets biometric-gate state. **No backup. No recovery. No undo.** The deliberate-confirmation interaction pattern (typed phrase or two-step tap, locked in file 11) is the architectural guardrail that makes this safe. **(c) Uninstall ≡ total deletion.** Android's app-uninstall semantics drop the app-private storage, which means both the Room database and WorkManager's own SQLite database are removed by the OS at uninstall; pending AlarmManager registrations are canceled by the OS when the owning app's package is removed. **Uninstall and the in-app hard-delete reach the same data-residue state (none)** — they differ only in user experience (uninstall removes the app icon too; hard-delete leaves the app installed and ready to start fresh). The **file 05 § Evidence five-year regulator-prudent retention tail** is a **different concept** that applies to the repository's audit artifacts (`_context/`, ADRs, handoff log, Privacy Policy versions, Play Console snapshots, dependency-review commit messages), **not** to the Patient's runtime data — the Patient's data lives on-device and is retained (or deleted) entirely at the Patient's discretion. A regulator asking for evidence about NemoPill's behavior gets the repo's audit trail; a regulator asking for evidence about *a Patient's* medication history gets nothing, by design, because nothing leaves the device.

- **Export or portability needs.** **No export feature in MVP — restated as a deliberate omission, not an oversight.** Exporting medication data to a file that could then be accidentally shared is itself a data-protection risk per file 05 § Data protection; file 01 keeps every Patient datum on-device by design. The CCPA / CPRA "right to know" obligation (file 05 § Compliance) is satisfied by the **in-app data-access surface itself** — every Medication, every Dose, and every Confirmation is readable from the in-app screens (Medication catalog, Today's Dose list, Adherence history) — and does not require an export affordance. The same in-app surface satisfies the equivalent rights under the Latin American national data-protection laws row (Mexico LFPDPPP, Argentina PDPL, Chile Ley 19.628, Colombia Ley 1581, Peru Ley 29733, Uruguay Ley 18.331, and the rest of the Spanish-speaking Play distribution targets). The **GDPR Article 20 portability** obligation that would force an export is **inactive at MVP** per file 05 § Compliance because Play distribution does not include the EU / EEA / UK; a future distribution expansion that includes any of those markets re-opens this rule as a fresh decision and likely forces the no-export rule to be revised. Adding an export feature in any other circumstance requires the file-05-defined revision path: (a) a revision of file 01's scope to permit Patient-driven sharing, (b) a Security-role review of the export format and the OS-level sharing surface, (c) a new ADR appended to `_context/09_decision_log.md`, and (d) a new row or revision in file 05 § Security Guardrails covering export-time encryption and the confirmation UX. **Symmetrically, there is no import feature in MVP either** — file 01's assumption (4) records that the Patient will manually configure their Medications and Dose Schedules without prescription import, and adding import follows the same revision path as adding export. The on-device-only posture is **bidirectionally enforced** at the architecture level (no `INTERNET` permission, no network-stack imports, no file-system-export affordance, no file-system-import affordance) so a future "let's add data sharing" conversation has to confront every layer of the commitment, not just one.
